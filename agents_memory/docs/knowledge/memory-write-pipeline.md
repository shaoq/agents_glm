## 0. 核心认知

记忆写入管线的本质，是把「**给人看的对话**」转成「**机器可高效召回的个性化记忆**」。三个关键判断：

1. **不是所有交互都值得记忆**：闲聊 / 寒暄 / 工具执行细节不应写入，只记可复用事实。
2. **写入质量决定召回上限**：写入的噪音 / 重复 / 矛盾，会直接污染后续召回（garbage in, garbage out）。
3. **写入是主动决策，不是被动存档**：每条候选记忆要经过「抽取 → 查重 → 冲突判断 → 存储」四步决策链。

---

## 1. 全景：写入管线的 4 个环节

```
交互消息 → ①事实抽取(Fact Extraction) → ②查重(Context Lookup)
        → ③冲突判断(ADD/UPDATE/NOOP) → ④存储(Memory Store)
```

一条主线：从原始对话到持久化记忆的完整流水线。下文逐环节展开。

---

## 2. 事实抽取（Fact Extraction）⭐

### 2.1 本质：为什么不直接存原文

直接存原始对话的问题：**噪音**（寒暄/过程细节淹没信号）、**不可复用**（「帮我看看代码」不可复用）、**检索效率低**（原文冗长稀释 embedding 信号）。抽取的本质是**从噪音中提取信号**——把「给人看的对话」提炼成「可被未来检索复用的独立事实」。

### 2.2 抽取什么 vs 不抽取什么

| 应抽取（可复用） | 不应抽取（噪音） |
|---|---|
| 用户偏好：「我喜欢用 Python」「回答要简洁」 | 寒暄：「你好」「谢谢」 |
| 稳定事实：「用户是后端开发者」「项目用 GLM-5.2」 | 过程细节：工具中间状态、临时变量 |
| 经验教训：「docling 解析中文 PDF 效果好」 | 临时信息：当前文件名、行号 |
| 决策/计划：「决定用 Chroma 起步」 | 无意义重复 |

**判断准则**：这条信息在未来交互中会被用到吗？会 → 抽取；不会 → 丢弃。

### 2.3 抽取粒度

| 粒度 | 做法 | 优 | 缺 |
|---|---|---|---|
| 粗 | 一条消息 → 一条摘要 | 快 | 含噪音 |
| 细 | 一条消息 → 多条独立事实 | 精确 | 可能碎片化 |
| **自适应**（推荐） | LLM 决定抽取几条（0-N） | 灵活 | 依赖 LLM 判断 |

### 2.4 类型分类：二分（fact / event）⭐

首版采用**二分**（而非三分 semantic/episodic/procedural）——三分边界模糊（semantic vs procedural 难区分），二分边界清晰：

| 类型 | 定义 | 时序性 | 可变性 | 例子 | 注入位置 | 维护策略 |
|---|---|---|---|---|---|---|
| **fact** | 跨时间稳定的事实/偏好/技能 | 无 | 可纠正（update） | 「用户偏好 Python」「PDF 用 docling」 | system prompt | 可纠正 |
| **event** | 特定时间发生的经历/经验 | 有 | 不可变，只能遗忘 | 「上次用户问了解析方案，建议了 docling」 | few-shot 参考 | 只能遗忘 |

**LLM 判断准则**（给 prompt 用）：这条记忆会随时间失去意义吗？是 → event（旧经历会过时）；否 → fact（稳定知识/偏好）。

**后续演进**：首版二分 → 如 procedural 需求明确 → fact 细分为 semantic + procedural。

### 2.5 重要性打分（importance）

写入时 LLM 同时打分（1-10），影响召回权重和遗忘淘汰：

| 分数 | 类型 | 例子 |
|---|---|---|
| 8-10 | 核心事实 | 用户身份、项目技术栈、关键决策 |
| 5-7 | 偏好/经验 | 回答风格偏好、某次操作经验 |
| 3-5 | 过程性 | 某次具体操作结果 |

### 2.6 抽取 prompt 设计

```
从以下对话中抽取可复用的事实和偏好。

要求：
1. 每条事实独立、自包含（脱离对话也能理解）
2. 客观陈述（「用户偏好 Python」），禁止主观推断（「可能不喜欢 Java」）
3. 一条一个要点，简洁
4. 不抽取：寒暄、过程细节、临时信息
5. 只抽取：偏好、稳定事实、经验教训、决策计划
6. 判断类型：fact（跨时间稳定）或 event（特定时间经历）

对话：{messages}

输出 JSON：
[{"content": "...", "type": "fact|event", "importance": 1-10}]
```

关键约束：客观陈述（禁推断）、独立自包含、类型 + importance 一次产出。

### 2.7 便宜模型适用性

抽取是「理解 + 结构化」任务，不需强推理/生成——**便宜模型（Flash 级）够用**。成本：每条交互 1 次 Flash（极便宜）。质量不足可升级主力（有成本）；需评测（准确率 / 噪音率）。

### 2.8 抽取时机

| 策略 | 做法 | 延迟 | 成本 |
|---|---|---|---|
| **实时**（推荐首版） | 每轮交互后抽取 | 低 | 每轮 1 LLM |
| 批量 | 累积 N 轮后批量 | 高（滞后） | 省 LLM |
| 混合 | 实时抽取关键 + 定期整理 | 低 | 适中 |

### 2.9 核心难点

| 难点 | 说明 |
|---|---|
| 抽取质量 | LLM 可能抽噪音或漏关键事实 → prompt 约束 + 评测 |
| 粒度控制 | 太粗含噪音 / 太细碎片化 → 自适应 + 评测 |
| importance 主观 | LLM 打分可能不准 → 需校准 |
| fact/event 边界 | 极少数情况模糊（经历沉淀为事实）→ 允许 LLM 判断，不强制 |

## 3. 查重（Context Lookup）⭐

### 3.1 本质：写入管线的门卫

查重是写入管线的**门卫环节**——在候选记忆进入冲突判断（§4）之前，先用便宜的方式判断「这条候选是否已有相似的」，只有查到相似才花 LLM 调用做精细判断。

**不查重的后果**：同一信息多次写入 → 记忆库膨胀 → 召回时同一事实重复出现 → 噪音污染。

### 3.2 查重 vs 冲突判断的职责分离

两步串联，**查重用便宜手段过滤，冲突判断用贵手段精判**：

```
候选记忆 → §3 查重（向量相似度，便宜：ms 级，无 LLM）
         ├─ 无相似 → 直接 ADD（跳过 §4，省 LLM）
         └─ 有相似 → 进入 §4 冲突判断（LLM 精判关系：重复/补充/矛盾）
```

**为什么两步而非一步**：如果每条候选都直接调 LLM 判断关系 → 候选多时成本高。先用向量相似度（ms 级、免费）过滤掉明确无相似的，只把「有相似」的交给 LLM 精判。

### 3.3 查重流程

```
候选记忆 content
  → 向量化
  → 在已有记忆库中检索 top_k（限制 user_id + type scope）
  → 如果 top1 相似度 > 阈值 → 进入 §4 冲突判断
  → 如果无相似 → 直接 ADD
```

### 3.4 相似度阈值

| 阈值 | 效果 | 风险 |
|------|------|------|
| 太高（0.95） | 只查几乎完全相同的 | 漏判：「用户喜欢 Python」vs「用户偏好 Python」语义同但分数低 |
| 太低（0.5） | 大量假阳性 | 每条候选都进 §4 LLM 冲突判断 → 成本高 |
| **经验值（0.7-0.8）** | 平衡 | 通用起点，需评测调优 |

### 3.5 跨类型查重

fact vs event 通常**不需要跨类型查重**：
- 「用户是 Python 开发者」（fact）vs「上次用户写了 Python 代码」（event）→ 语义相似但不是重复
- fact 是稳定知识，event 是时序经历——**语义相似 ≠ 重复**
- 查重只在**同类型内**进行（fact 查 fact、event 查 event）

### 3.6 性能特性

- 每条候选记忆查一次已有库（向量查询 ms 级）
- 一次交互抽 3-5 条候选 → 3-5 次查询（可接受）
- user_id + type scope 过滤缩小检索范围（不全库扫描）
- 批量优化空间：多条候选向量化后批量查询

### 3.7 核心难点

| 难点 | 说明 |
|------|------|
| **阈值调参** | 太高漏判（重复写入）、太低假阳性（浪费 LLM 冲突判断）。需评测 |
| **语义等价 vs 表述不同** | 「喜欢 Python」vs「偏好 Python」→ embedding 相似度可能不够高 → 漏判 |
| **首次写入空库** | 记忆库初始为空 → 查重无结果 → 全部 ADD（正常冷启动） |
| **同义不同词** | 「GLM-4.5」vs「智谱大模型」→ 向量可能区分不了（取决于 embedding 模型） |

### 3.8 常见陷阱

| 陷阱 | 后果 |
|------|------|
| 全库查重（不 scope user_id） | 跨用户误判相似 → 合并不同用户记忆 |
| 跨类型查重 | fact vs event 语义相似但非重复 → 误进冲突判断 |
| 阈值拍脑袋 | 太松（重复堆积）或太紧（大量 LLM 冲突判断） |

> **核心认知**：查重的价值不在判断精确性（那是 §4 的事），而在**用最低成本排除明确无相似的候选**，让 §4 只处理真正需要精判的少数情况。
