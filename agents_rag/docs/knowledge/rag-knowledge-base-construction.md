## 0. 核心认知

知识库构建的本质，是把**「给人看的文档」转成「机器可高效检索、可溯源、低噪声的语义索引」**。在 RAG 体系里这条线叫 **索引管线（indexing pipeline）**，是离线 / 批处理过程。

三个关键判断：

1. **质量决定上限**：查询侧（检索、重排、生成）再精巧，也救不了一个构建得烂的知识库（garbage in, garbage out）。
2. **70% 法则**：工作量与质量风险，约 **70% 集中在「文档解析」和「分块」两步**；向量化反而是最标准化的一步。
3. **三段总纲**：
   - **解析**决定「信息能不能拿出来」
   - **分块**决定「信息能不能被精准召回」
   - **元数据**决定「能不能溯源回去」

三者缺一不可。

## 3. 文档解析（质量上限的瓶颈）⭐⭐⭐

知识库构建里**最难、最影响上限**的一步。解析不好，后面全是无米之炊。功夫的 80% 花在 PDF 上。

### 3.1 本质：一次「呈现层 → 语义层」的翻译

所有文档格式都是为**给人看 / 给渲染器画**设计的，不是为机器理解。解析的本质，是把「视觉版面」翻译成「结构化语义」：

```
PDF/Office/HTML（呈现层：坐标、样式、标签）
        ↓ 解析
Document（语义层：标题层级、段落、表格、列表 + 位置元数据）
```

不同格式的「语义保留度」天差地别，决定难度：

| 格式 | 语义保留度 | 难度 |
|------|-----------|------|
| Markdown | 高（标题/列表是显式语法） | 极低 |
| HTML | 中高（有标签，但混导航/广告噪音） | 低-中 |
| Office（docx/xlsx/pptx） | 中（有结构标签，但版式干扰） | 中 |
| **PDF** | **低（只有坐标，无语义）** | **高** |
| 扫描件 PDF | 零（只有像素） | 极高（要 OCR） |

PDF 是地狱难度，也是企业文档的大头，所以解析功夫 80% 花在 PDF 上。

### 3.2 PDF 解析深水区

**PDF 内部有什么**：一组**绘制指令**（在坐标 (x,y) 处画字符）。它没有「段落 / 标题 / 表格」概念——这些是人脑补出来的。所以解析是**反向工程**：从带坐标的字，重建人看到的结构。

**三代解析技术的演进**：

| 代 | 代表 | 方法 | 特点 |
|----|------|------|------|
| 第一代 | pdfplumber / PyMuPDF | 规则 + 启发式（坐标聚类成行/段） | 快、可控，但脆弱，复杂版面崩 |
| 第二代 | LayoutParser / deepdoctection | 检测模型找区域 + 规则重组 | 版面识别准，但表格/顺序仍弱 |
| 第三代 | **Docling / MinerU / Nougat** | 端到端文档理解模型（视觉 + 结构联合） | 表格、公式、多栏都强，当前主流 |

> agents_rag 选 Docling/MinerU 即因它们是第三代，能力代差明显。

**PDF 解析要解的 5 个子问题**：

- **a. 版面分析（Layout Analysis）**：把页面切成区域（文本块 / 标题 / 表格 / 图片 / 公式 / 页眉页脚），底层是目标检测模型。决定每块「是什么类型」。
- **b. 阅读顺序还原**：多栏文档，坐标排序会把左右栏混排；要识别「栏」，再按「栏内自上而下、栏间自左而右」排序，跨页还要处理段落承接。
- **c. 表格识别（最值钱）**：两步——**结构识别**（几行几列、合并单元格）+ **内容提取**。差的拆成散字，好的输出结构化（行列矩阵 / Markdown 表）。对企业财务/法规文档决定可用性。
- **d. 公式识别**（学术 PDF）：把公式图片转 LaTeX，MinerU/Nougat 强项。
- **e. OCR（扫描件）**：PDF 无文字层时，先**文字检测**（定位）再**文字识别**（认字）。RapidOCR/PaddleOCR 主流，中文友好，CPU 可跑。

**OCR 触发判断**：解析后若**文本量异常少**（如每页 < N 字符），判定为扫描件，自动转 OCR 路径。

### 3.3 其他格式（相对简单）

- **Office**：结构化程度好。`python-docx` 读 Word（段落 + 标题样式 + 表格），`openpyxl` 读 Excel（直接行列），`python-pptx` 读 PPT。难点是样式 → 语义映射（「加粗大字」算不算标题）。
- **HTML**：`trafilatura` 直接抽正文（去导航 / 广告 / 侧栏），`BeautifulSoup` 兜底清洗。
- **Markdown**：天然结构化，按标题 / 段落解析，最省心。

### 3.4 统一输出模型（关键设计）⭐

不同 parser 输出各异，**必须统一到一个内部 `Document` 结构**，下游（分块 / 索引）才能不关心来源格式：

```
Document
├── doc_id, source, doc_type
├── sections: list[Section]        # 按标题层级组织的章节树
│   └── Section
│       ├── heading, level         # 标题 + 层级（H1/H2…）
│       ├── blocks: list[Block]    # 段落 / 表格 / 列表
│       │   └── Block(type, text, page, bbox, table_data?)
│       └── children: list[Section]
└── metadata
```

设计要点：

- **类型化 Block**（paragraph / table / list）：分块时按类型不同处理（表格不能切断）
- **保留位置**（page、bbox）——引用溯源的根
- **章节树**——结构感知分块的输入

这一层做不好，后面分块会很别扭。

### 3.5 工具选型对比

| 工具 | 强项 | 弱项 |
|------|------|------|
| **Docling** | 输出结构化（标题/段落/表格带类型），表格识别强，输出 Markdown/JSON | 极复杂学术 PDF 偶有遗漏 |
| **MinerU** | 复杂中文/学术 PDF 极强（公式、双栏、表格），保留版面 | 依赖较重（带模型） |
| Unstructured | 全格式通用、生态广 | 表格一般，中文 PDF 不如前两者 |
| PyMuPDF / pdfplumber | 轻量、可控 | 需自己处理结构逻辑 |

### 3.6 降级链与路由（工程关键）⭐

`ParserRouter` 是解析层的调度中心：

```
按扩展名选主 parser
   ↓
解析 → 评估质量（文本量/表格完整度）
   ↓ 质量差
降级到备选（如 PDF→MinerU；文本过少→OCR）
   ↓ 仍失败
记录日志、跳过该文件（不中断批量）
```

原则：**永远要有兜底，单文件失败不能拖垮整批索引**。

### 3.7 解析质量评估

经验信号：每页平均字符数（异常低 → 可能扫描件 / 解析失败）、表格识别比例、标题层级分布、乱码 / 空白占比。可做成解析后自动检查，质量差的文件告警人工复核。

### 3.8 agents_rag 落地

```
file → ParserRouter
        ├─ .pdf  → Docling → (质量差?) → MinerU → (文本少?) → RapidOCR
        ├─ .docx → python-docx
        ├─ .xlsx → openpyxl
        ├─ .pptx → python-pptx
        ├─ .html → trafilatura + BeautifulSoup
        └─ .md   → markdown 解析
        ↓
   统一 Document(sections + blocks + 位置元数据)
```

### 3.9 解析层常见陷阱

| 陷阱 | 后果 |
|------|------|
| 用第一代规则解析器啃复杂 PDF | 表格 / 多栏全崩 |
| 不做 OCR 兜底 | 扫描件直接丢字 |
| parser 输出不统一 | 下游分块要为每种格式写特例 |
| 不保留 page/bbox | 无法引用溯源 |
| 表格被当普通文本切 | 表格数据损毁 |
| 单文件失败中断整批 | 索引管线脆弱 |

**核心 takeaway**：解析是「呈现 → 语义」的翻译，PDF 是主战场，靠**第三代文档理解模型 + 统一输出模型 + 降级链兜底**。

## 4. 清洗归一化（净化层）⭐

解析之后、分块之前。把解析输出的「带噪原始内容」整理成**干净、规范、适合分块与 embedding** 的形式。它是「承上（解析）启下（分块）」的净化层。

「轻」不等于「可省」——脏数据进去，后面 chunk 和 embedding 全被污染。

### 4.1 为什么需要

解析输出的内容天然带噪：

- **页眉页脚每页重复**（「第 X 页 | 公司名 | 章节名」）——同一噪声被切成几十个 chunk
- **水印、批注、修订标记**
- **PDF 硬换行**导致碎句（一句话被切成几行）
- 多余空白、空行
- 编码 / 乱码、特殊字符
- 全半角标点混用
- OCR 错字、孤立字符

不清这些噪声，会**稀释 embedding 信号、制造无效 chunk、干扰检索排序**。

### 4.2 清洗的六类操作

| 类别 | 做什么 |
|------|--------|
| **去噪** | 页眉页脚、页码、水印、目录、脚注引用标记 |
| **空白规整** | 合并 PDF 断行、去多余空格 / 空行 |
| **编码归一** | 统一 UTF-8、修常见乱码 |
| **标点归一** | 全半角统一、中文标点规范 |
| **结构清洗** | 去空标题、合并碎片段 |
| **OCR 后处理**（可选） | 规则或小模型纠正常见错字 |

### 4.3 关键原则：克制度 ⭐

清洗最忌「用力过猛」：

- **去噪 ≠ 误删正文**：表格里的编号、公式符号、代码缩进不是噪声，删了就丢信息
- **只动文本层，不破坏结构**：解析得到的标题层级、表格、列表要保留
- **保留位置元数据**（page / bbox）——溯源根基不能在清洗时丢
- **可追溯**：记录清洗前后的差异，便于回溯调试

### 4.4 边界：和解析、分块怎么分工

- **依赖解析**：去页眉页脚其实可以在解析阶段用版面分析直接做掉（识别出「页眉区」丢弃）。清洗与解析的边界是灵活的。
- **服务分块**：清洗的验收标准是——「分块拿到的是干净的」。
- 实践中：**版面级去噪归解析，文本级规整归清洗**。

### 4.5 常见陷阱

| 陷阱 | 后果 |
|------|------|
| 过度清洗 | 删掉有用信息（编号、符号、缩进） |
| 一刀切规则 | 不同文档类型需要不同清洗规则 |
| 不留原文 | 出问题无法回溯 |
| 清洗时丢元数据 | 引用溯源断链 |

### 4.6 agents_rag 落地

- 作为 pipeline 的独立步骤（解析后、分块前），或解析的后处理
- 清洗规则 **按 doc_type 可配置**（PDF / HTML / Office 规则不同）
- 保留 page / heading 等元数据
- 记录清洗日志，可追溯

**核心 takeaway**：清洗是克制地去噪——**不动结构、保留元数据、可追溯**。

## 5. 分块（Chunking）⭐⭐⭐ —— 知识库构建的核心技术

如果说解析是「拿出来」，分块就是「切得对」。**这是知识库质量最关键的杠杆**。

### 5.1 本质再认识：chunk 是「检索单元」不是「阅读单元」

最关键的认知转变。很多人按「给人读的段落」来切 chunk，是错的。chunk 的设计目标是**检索效率**：让相关语义能被精准命中。它不需要自带完整上下文（上下文可靠父块 / 窗口回传补足）。这个认知一旦建立，就理解了为什么子块可以切得很碎、chunk 可以不带完整上下文、父子分块与句子窗口这些「分离粒度」设计才合理。

### 5.2 为什么要分块（核心矛盾）

- **检索粒度**：整篇文档做向量，召回时信号被稀释（一个向量代表几万字，必然模糊）
- **上下文预算**：LLM 输入有限，要精准喂相关片段
- **召回精度**：块越小越精确，但太小会丢上下文 → **核心矛盾**

> 经验法则：chunk 太大 → 噪声多、token 浪费、召回稀释；chunk 太小 → 语义残缺、答案缺上下文。**没有银弹，要按数据调**。

### 5.3 分块策略（从简单到高级）

**① 固定大小分块**：按 token / 字符数硬切 + overlap。最简单，但会从句子/表格中间切断，基线可用、质量差。

**② 递归字符分块**：按分隔符优先级切（`\n\n` 段落 → `\n` → `。` 句 → 空格）。主流基线，比固定好，但仍忽略文档结构。

**③ 结构感知分块**（推荐）：利用解析得到的标题层级，按「章节 / 小节 / 段落」自然边界切。一块属于一个语义单元，质量最高。

**④ 语义分块**：用 embedding 相邻句相似度找「语义断点」。自适应，但要预计算 embedding，慢、成本高。

**⑤ 格式感知**：表格、列表、代码块必须整体保留，绝不在内部切断。常被忽略，但对表格密集的领域文档（财务、法规）是致命的。

### 5.4 结构感知分块：算法实现

输入是解析得到的章节树（§3.4 的 `Document → sections → blocks`）：

```
遍历章节树:
  对每个 section:
    收集其下 blocks（段落/表格/列表）
    累加 block 文本:
      if 累加长度 ≤ chunk_size:   继续聚合
      else:                        切出一个 chunk，开启新 chunk
    特殊 block（表格/代码/列表）:  整体作为一个 chunk，不参与累加切断
  每个 chunk 继承: section_path（章节路径）, page, heading
```

三个要点：

- **尊重 block 边界**：切断只发生在 block 之间，绝不切断一个段落 / 表格
- **标题继承**：chunk 带 `section_path`（如「第三章 > 3.2 分块 > 算法」），检索命中后提供「这个片段在讲什么」的上下文
- **特殊结构豁免**：表格 / 代码 / 列表作为原子块，再大也整体保留

### 5.5 父子分块：工程实现 ⭐

**数据结构**：

```
子块 child_chunk:  id, parent_id, text(小), embedding, page, heading
父块 parent_chunk: id, text(大, 含完整段落/小节), page, heading
```

**存储（双索引）**：向量索引只建在**子块**上（子块负责被检索）；父块单独存一个 KV（parent_id → 父块文本），不建向量索引。

**查询流程**：

```
query → 向量检索子块 top_k → 得到命中的 child_ids
      → 取 parent_ids（去重）
      → 查 KV 取父块文本
      → 父块回传给 LLM
```

**进阶：AutoMerging（自动合并）**：当一个父块下的**多个子块都被命中**，说明这一整段高度相关——直接回传整个父块，而非返回多个重叠片段。避免回传内容冗余和截断。LlamaIndex 的 `HierarchicalNodeParser` 即此思路。

**变体：句子窗口**：不显式建父子，而是检索单句、回传时用「该句 ± N 句」拼出窗口。实现更轻，但上下文控制不如父子灵活。

> **agents_rag 选型**：父子分块（结构感知切父块 → 父块再切子块 → 子块建索引）。比句子窗口更适合有清晰结构的领域文档。

### 5.6 调参方法论（最重要、最被忽视）⭐

**核心原则：chunk_size 不是拍脑袋定的，是评测出来的。** 概览给的「中文 300–600 字」是范围，不是答案。调参流程：

1. 建小评测集（30–50 条 query + 每条「应被召回的文档」）
2. 选一组候选 chunk_size（如 256 / 384 / 512 / 768），相同 overlap 比例
3. 每组建索引，跑评测集，记 **recall@k** 和**答案质量**
4. 选 recall 最高且 chunk 语义完整的那个

**联动关系（关键）**：

- **chunk_size ↔ top_k**：块越小，单个块信息越少，要多召回几个（top_k 大）；块越大，少召回几个。两者一起调。
- **chunk_size ↔ overlap**：overlap 防边界丢失，但过大则存储 / 计算冗余。一般 10–20%。
- **chunk_size ↔ embedding 上下文**：embedding-3 单条上限 3072 token，块别超（远不会超）。

**经验锚点**（起点，非终点）：

- 事实 / 定义密集（法规、手册）：偏小（300–400 字），精准
- 叙述 / 论述密集（报告、论文）：偏大（500–700 字），保上下文

### 5.7 特殊结构处理（领域文档的命门）

| 结构 | 处理 | 原因 |
|------|------|------|
| **表格** | 整体一个 chunk，转 Markdown 表 | 切断 = 行列关系崩 |
| **列表** | 整体保留 | 切断 = 条目脱离语境 |
| **代码块** | 整体保留，保缩进 | 切断 = 语法 / 逻辑断 |
| **公式** | 整体保留 | 切断 = 不可读 |

表格密集的领域（财务报表、产品规格），这步做不好整张表变乱码，前面解析再好也白搭。

### 5.8 进阶：层级索引（多粒度）

同一文档切**多个粒度**（如 256 / 512 / 1024 三层），建多套索引。查询时：简单事实查询 → 小粒度（精准）；综述类查询 → 大粒度（完整上下文）；用「查询路由器」判断走哪层。成本高（多套索引），适合查询类型多样的场景。agents_rag 第一版用单层父子即可，这是后续优化方向。

### 5.9 中文分块细节

- **不按裸字符数切**：会切断词（如「机器」切成「机」/「器」）
- **按标点 / 段落优先级切**：`。！？\n\n` → `\n` → 逗号
- **token vs 字符**：embedding 按 token 计费，中文 1 字 ≈ 1–2 token；chunk_size 设定时注意是「字符」还是「token」口径
- **jieba 的作用**：主要在 BM25 侧；分块侧用标点边界足够

### 5.10 分块质量评估指标

几个可量化信号：

- **平均 chunk 长度分布**：是否集中在预期区间，有无大量极短 / 极长块
- **截断率**：在句子 / 表格内部被切断的 chunk 占比（应趋近 0）
- **语义完整度抽检**：人工看一批，是否语义自洽
- **召回测试**：recall@k 是最终评判

### 5.11 元数据附着（每个 chunk 必带）

```
doc_id | source | page | heading | chunk_idx | parent_id | section_path
```

**没有元数据 = 无法引用溯源 = 企业场景不可用。**

### 5.12 agents_rag 落地

```
Document(章节树)
   → 结构感知切父块 (尊重 block 边界, 表格/代码豁免)
   → 父块再切子块 (子块粒度由 chunk_size 控制)
   → 子块带 parent_id + section_path + page + heading
   → 子块建向量索引; 父块存 KV
   (查询: 子块命中 → AutoMerging 取父块回传)
```

参数（chunk_size / overlap / top_k）全部配置化，配评测集调优。

### 5.13 分块层常见陷阱

| 陷阱 | 后果 |
|------|------|
| 照抄默认 chunk_size | 不适配你的数据 |
| 只用固定大小分块 | 切坏句子 / 表格 |
| 表格 / 代码被切断 | 结构数据损毁 |
| 检索粒度=回传粒度（无父子） | 要么召回稀释，要么上下文不全 |
| 不调参、不评测 | 质量靠玄学 |
| overlap 过大 | 冗余、成本高 |
| chunk 不带 section_path | 命中后缺上下文定位 |

**核心 takeaway**：分块的本质是把文档切成「最佳检索单元」——靠**结构感知切分 + 父子分离粒度 + 评测驱动调参 + 特殊结构豁免**四件套。其中**调参方法论是最该建立的能力**：别抄参数，用评测集调。

## 6. 向量化与索引（最标准的一步）⭐⭐

向量化是「语义召回」，BM25 是「精确词召回」，两者互补才是完整召回。这一步原理标准化，但工程化（并发 / 缓存 / 双索引同步）有讲究。

### 6.1 Embedding 原理：为什么向量能做语义检索

文本经 embedding 模型映射成**高维向量**（embedding-3 是 2048 维）。关键性质：**语义相近的文本，向量在空间中距离近**。

```
"如何重置密码"      ──→  [0.12, -0.33, ...]  ┐
"忘记密码怎么办"    ──→  [0.11, -0.31, ...]  ├─ 距离近（语义同）
"密码重置流程"      ──→  [0.13, -0.35, ...]  ┘
"今天天气不错"      ──→  [0.85, 0.02, ...]   ── 距离远（语义不同）
```

这是关键词检索做不到的——「重置密码」和「忘记密码」没有共同关键词，但语义一样，向量能匹配。

**bi-encoder 架构**：query 和 doc **独立编码**成向量。doc 可离线编码建库，查询时只编码 query 一次，再和库里所有 doc 向量算距离。这是向量检索快的基础（也是精度上限的来源——查询侧 rerank 的 cross-encoder 才互补）。

### 6.2 相似度度量

| 度量 | 说明 | RAG 用法 |
|------|------|---------|
| **余弦相似度** | 比向量方向（归一化后），忽略长度 | **默认首选** |
| 点积 | 未归一化的余弦，向量长度影响结果 | 归一化向量下等价余弦 |
| 欧氏距离 | 几何距离 | 少用 |

为什么默认余弦：文本 embedding 的「语义强度」不应被向量长度干扰，余弦只看方向（语义朝向），最稳。

### 6.3 向量索引算法（ANN）⭐ —— 工程核心

库一大，暴力比对所有向量（Flat）就慢了。于是有**近似最近邻（ANN）**算法，用一点精度换巨大速度提升。

| 算法 | 原理 | 特点 |
|------|------|------|
| **Flat（暴力）** | 逐一算距离 | 精确，但 O(N)，仅小库可用 |
| **HNSW** | 分层小世界图，导航式逼近 | 查询快、召回高，**主流**；内存高、构建慢 |
| **IVF** | 先 K-means 聚类，查询只扫几个簇 | 可调探测数（nprobe）平衡速度/召回 |
| **PQ（乘积量化）** | 向量压缩省内存 | 省内存但损精度，常和 IVF 组合（IVF-PQ） |

**关键权衡三角**：召回率 ↔ 查询速度 ↔ 内存占用，没有全占。

> agents_rag 用 Chroma，默认 HNSW——中小规模 + 本地优先场景的最佳选择。规模化时 HNSW 的内存开销才需关注。

### 6.4 Embedding 批处理与并发（工程化）⭐

embedding 是付费 API 调用，工程化要点：

- **批处理**：单次最多 64 条（embedding-3 限制）、单条 ≤ 3072 tokens，批量摊薄网络开销、降本提速
- **维度权衡**：2048 最高精度（领域文档推荐）/ 1024 平衡 / 512 省
- **并发控制**：信号量限在途请求数（受账号等级限流：V0=50 … V3=500），超了 429
- **重试**：tenacity 指数退避，区分可重试（429/5xx）与不可重试（鉴权错）
- **进度持久化**：批量时记录「已处理到哪」，中断可续跑（别白花钱重算）
- **异步**：aiohttp / asyncio 并发请求，别串行等

### 6.5 Embedding 缓存策略 ⭐

**最被低估的省钱/提速手段**。同一块文本（如未变更 chunk）不应重复 embed。

- **缓存键**：文本的 hash（SHA-256），文本不变则向量不变
- **存储**：本地 KV（sqlite / 磁盘文件）
- **复用场景**：增量索引时未变更文档的 chunk 命中缓存；调参重跑时相同 chunk 复用
- **收益**：大幅降本 + 实验可复现（同样输入同样向量）

> 没有缓存的 RAG，每次调参重索引都全量付费重 embed——成本和迭代速度双重打击。

### 6.6 BM25 原理与中文适配 ⭐

**为什么需要它**：向量对「型号、编号、人名、专有术语」召回差（这些词语义向量不够区分）。BM25 是经典全文检索算法，专攻精确词匹配。

**BM25 是什么**：TF-IDF 的概率论改进版。直觉：

```
score(query, doc) = Σ 对每个 query 词:
    IDF(词) × TF饱和(词,doc) × 文档长度归一化
```

- **IDF**：词在全局越稀有权重越高（「的」低，「GLM-4.5」高）
- **TF 饱和**：词频贡献有上限（10 次不比 5 次好两倍）
- **文档长度归一化**：长文档不因「字多」占便宜

**中文适配（关键）**：BM25 建立在「词」上，中文无空格分词，**必须先分词**（jieba），否则按单字 BM25 效果差、噪声大。

```
原文：智谱GLM4.5模型
jieba：智谱 / GLM / 4.5 / 模型  →  BM25 对「词」建索引
```

### 6.7 双索引的存储与一致性 ⭐

向量索引和 BM25 索引是两套独立存储，**必须保持一致**：

- **ID 对齐**：两边用同一个 `chunk_id`，查询结果能对应回同一块
- **同步增删改**：文档变更时，向量库 upsert + BM25 重建/增量，不能只更一边
- **rank_bm25 持久化**：默认内存型，要自己 pickle 到磁盘（或用 bm25s 支持磁盘）

### 6.8 向量库选型与迁移路径

| 向量库 | 定位 | 适用 |
|--------|------|------|
| **Chroma** | 嵌入式、零运维 | 起步 / 中小规模 / 本地（agents_rag 起步选） |
| **Qdrant** | 高性能、强过滤、可服务化 | 中大规模、生产 |
| **Milvus** | 分布式 | 大规模、海量 |
| **FAISS** | 库（非服务） | 要自己包一层服务 |

**迁移路径（先小后大的关键）**：抽象 `VectorStore` 接口（upsert/delete/query），初期 Chroma，规模化时换 Qdrant——业务代码不动。

### 6.9 质量评估

- **embedding 健康度**：相关对 vs 不相关对的相似度分布是否分离（分得开才好用）
- **ANN 召回率**：ANN 结果 vs 暴力检索 ground truth 的重叠率（recall@k）
- **向量 vs BM25 召回对比**：看两类各自召回什么，验证互补性

### 6.10 agents_rag 落地

```
子块文本 → (查缓存) → 批量 embedding-3 (并发限流 + 重试) → 写缓存
        → Chroma 向量库 (HNSW) [VectorStore 抽象]
        → BM25 索引 (jieba + rank_bm25, 持久化)
两索引以 chunk_id 对齐, 增删改同步
```

### 6.11 向量化与索引层常见陷阱

| 陷阱 | 后果 |
|------|------|
| 只建向量索引，无 BM25 | 专有名词 / 编号召回漏 |
| 无 embedding 缓存 | 调参重跑全量付费、不可复现 |
| 并发不限流 | 频繁 429、批量中断 |
| 中文 BM25 不分词 | 按字匹配、噪声大 |
| 双索引不同步 | 召回结果对不上 |
| 向量库未抽象接口 | 规模化时迁移痛 |
| 忽略 ANN 召回率 | 看似快，实际漏召回 |

**核心 takeaway**：向量化是「语义召回」，BM25 是「精确词召回」，**两者互补才是完整召回**；工程上靠**批处理 + 并发限流 + 缓存 + 双索引同步**把这一层做稳。索引算法选 HNSW（Chroma 默认），向量库抽象接口为规模化留路。

## 7. 元数据与（可选）知识图谱

- **元数据**：来源、时间、作者、章节路径、页码、权限标签。是**过滤检索、引用溯源、权限控制**的基础。
- **知识图谱**（高级，非第一版目标）：从文档抽取实体 / 关系，支持结构化推理与多跳检索。复杂领域（医疗、法律）收益大，但工程重。

---

## 8. 持久化与增量维护（运维关键）

- **文档指纹表**：`doc_id(fingerprint) → 状态 / 索引位置`，持久化
- **增量 upsert**：只重算变更文档，不全量重建
- **删除同步**：源文件消失则清理其所有 chunk 与向量
- **版本 / 时间戳**：支持文档多版本与回溯

> **agents_rag 落点**：维护一份指纹清单文件，索引管线启动时 diff 出「新增 / 更新 / 删除」三类，分别处理。这是「先小后大」能扩到中规模的前提。

---

## 9. 知识库质量保障（最常被忽视）

知识库构建不是「跑一次就完」，要有评估闭环：

- **分块抽检**：人工看一批 chunk 是否语义完整、无截断
- **召回测试集**：准备一批 query + 「应被召回的文档 / chunk」，量化召回率（recall@k）
- **embedding 健康度**：相关 / 不相关对的相似度分布是否合理
- **参数迭代**：chunk_size、overlap、top_k 反复调
- 与查询侧的 **RAGAS 评估互补**：RAGAS 评「答得好不好」，召回测试评「找得准不准」

## 10. 常见陷阱清单（避坑）

| 陷阱 | 后果 |
|------|------|
| 一刀切固定大小分块 | 切坏句子 / 表格，召回残缺 |
| 忽略表格 / 结构 | 表格变乱码，领域数据丢失 |
| 不带元数据 | 无法引用溯源 |
| 只建向量索引 | 专有名词 / 编号召回差 |
| 不做增量 | 数据一变就全量重建，慢且贵 |
| embedding 无缓存 | 重复计算，成本高、不可复现 |
| 解析质量差 | 上限被锁死，下游再优化也无效 |
| 过度清洗 | 删掉有用信息（编号、符号、缩进） |

## 11. 小结：知识库构建配方（agents_rag）

```
data/raw/ (指纹 diff)
   → ParserRouter (Docling → MinerU → OCR 降级)
   → 清洗归一化
   → 结构感知分块 + 父子分块 (带 page/heading/parent_id)
   → embedding-3 批量向量化 (缓存 + 限流)
   → Chroma 向量库 + BM25 索引 (双索引)
   → 指纹清单持久化 (支持增量)
   → 召回测试集 + 抽检 (质量闭环)
```

核心取舍：**在「解析」和「分块（父子）」上投入最多**（质量杠杆）；向量和检索用成熟方案标准化。
