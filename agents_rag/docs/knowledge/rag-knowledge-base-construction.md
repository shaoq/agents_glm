## 0. 核心认知

知识库构建的本质，是把**「给人看的文档」转成「机器可高效检索、可溯源、低噪声的语义索引」**。在 RAG 体系里这条线叫 **索引管线（indexing pipeline）**，是离线 / 批处理过程。

三个关键判断：

1. **质量决定上限**：查询侧（检索、重排、生成）再精巧，也救不了一个构建得烂的知识库（garbage in, garbage out）。
2. **70% 法则**：工作量与质量风险，约 **70% 集中在「文档解析」和「分块」两步**；向量化反而是最标准化的一步。
3. **三段总纲**：
   - **解析**决定「信息能不能拿出来」
   - **分块**决定「信息能不能被精准召回」
   - **元数据**决定「能不能溯源回去」

三者缺一不可。

## 2. 采集与接入（入口）⭐

采集是构建管线的入口，核心不只是「读文件」，而是**增量检测的健壮性**——决定数据变了能不能低成本、正确地更新。

### 2.1 数据源与格式

- **数据源**：本地文件 / 对象存储 / 数据库 / 网页爬取 / 内部 API
- **格式清单**：PDF、Office（Word/Excel/PPT）、HTML、Markdown、邮件、代码……

### 2.2 身份认定：内容指纹为锚

增量检测的前提是回答「这是不是同一个文档」。两种身份依据：

| 身份依据 | 稳定性 | 问题 |
|---------|--------|------|
| 路径 / 文件名 | 差（文件会移动、改名） | 一移动就误判为新文档 |
| **内容指纹（SHA-256）** | **好（内容不变则不变）** | 改一个字就是新身份 |

agents_rag 用**内容指纹为主键**，带来几个要处理好的边界（见 §2.5）。

### 2.3 核心数据结构：文档注册表 ⭐

增量检测的数据结构分两层，**文档注册表是 source of truth**：

```python
# 文档注册表（持久化 sqlite/json）—— 真相源
DocumentRecord:
    doc_id: str               # = content_fingerprint + namespace，主键
    content_fingerprint: str  # SHA-256(content)
    source_path: str          # 当前路径（移动时更新）
    source_namespace: str     # local / oss / crawl ...
    doc_type: str
    chunk_ids: list[str]      # ★ 该文档所有子块 id（删除/更新的精确句柄）
    parent_chunk_ids: list[str]
    version: int
    acl: list[str]
    content_size: int         # 预筛 + 校验
    indexed_at: datetime
    status: str               # active / deleted
```

块级存储（向量库 / BM25 / 父块 KV）按 `chunk_id` 存内容。三者关系：

- 注册表 = 目录（哪些文档已索引、chunk_ids 在哪）
- 内容存储 = 按 chunk_id 存
- `chunk_ids` 是「注册表 ↔ 内容存储」的桥梁——删除 / 更新靠它精确操作，没有它就没法干净地删改

> 关键设计：**doc_id = 指纹 + namespace，chunk_id 全局唯一，注册表持有 chunk_ids 反查表**。同时支撑五态检测、下游更新、引用溯源。

### 2.4 大文件哈希：流式 + 预筛 + 并行

整个大文件读进内存算 hash 会爆内存。标准解法：

**a. 流式 SHA-256（恒定内存，必做）**——增量算法，分块 update 与一次性算结果相同：

```python
import hashlib
def file_hash(path, buf=1 << 20):      # 1MB buffer
    h = hashlib.sha256()
    with open(path, 'rb') as f:
        while chunk := f.read(buf):
            h.update(chunk)
    return h.hexdigest()
```

内存恒定（1MB + hash 状态），文件再大也不怕。

**b. (size, mtime) 预筛（跳过未变文件）**——先用便宜的 (size, mtime) 判断「可能没变」，一致就跳过精确 hash；变了才算 hash 确认。预筛是「可能变了」，hash 是「确实变了」，不会漏判。

**c. 多文件并行**——hash 是 IO 密集，线程 / 进程池并行。

> ⚠️ 不推荐「部分 hash」（头尾 + 采样）：内容变在没采样处会漏检。RAG 增量必须准确，漏检 = 脏数据。

### 2.5 五态检测与边界 ⭐

以内容指纹为身份后，diff 从「四态」扩展为「**五态**」：

| 状态 | 判定 | 处理 |
|------|------|------|
| **新增** | 指纹表中无 | 完整索引 |
| **更新** | 指纹在表但内容变 | 删旧 + 重新索引 |
| **删除** | 表中有、源没了 | 清理所有 chunk |
| **移动** | 指纹已在表、但路径变 | **只更 source_path，不重索引** |
| **未变** | 指纹相同 | 跳过 |

边界处理：

- **复制 vs 移动**（同指纹两路径）：看旧路径是否还在——旧没了是移动（更路径），都在是复制（按策略去重或都留）
- **多源同名**：本地 / 对象存储 / 爬取的同名文件内容可能不同 → `doc_id = 指纹 + namespace`，绝不按文件名去重
- **哈希碰撞**：SHA-256 ≈ 0，可忽略
- **纯元数据 / 权限变化**（内容没变）：只更元数据，不重索引
- **大文件局部更新**：仍文档级重索引（块级 diff 成本太高）

### 2.6 五态 → 动作：驱动下游正确更新 ⭐⭐

检测出五态后，翻译成精确到 chunk_id 的动作（Action）：

| 状态 | 下游动作 | 索引操作 |
|------|---------|---------|
| 新增 | 完整流水线 | 解析→分块→embedding→写三索引 → 注册表 insert |
| 更新 | 先建新后删旧 | 新版流水线得新 chunk_ids → 删旧 chunk_ids → 注册表 update(version++) |
| 删除 | 清理 | 按 chunk_ids 删三索引 + 缓存 → 注册表 delete |
| 移动 | 仅更元数据 | 注册表 update(source_path)，索引不动 |
| 未变 | 跳过 | 无操作 |

**两阶段执行**（避免状态不一致）：先扫描全部产出 action list，再批量执行。

```python
actions: list[Action] = diff(scan_result(), registry)   # 阶段1：检测
for action in actions:
    apply(action)                                        # 阶段2：执行
```

**保证正确更新的要点**：

1. **chunk_id 精确句柄**：删改靠注册表的 chunk_ids，不靠「按 doc 模糊删」
2. **先建新后删旧**：新 version 区分，先写新确认成功再删旧，杜绝空窗丢数据
3. **多索引原子**：向量库 + BM25 + 父块 KV + 缓存一起成功或回滚，记操作日志可重放
4. **幂等**：new 前查存在性、delete 不存在不报错，失败重试安全
5. **顺序**：先 new / update 再 delete，避免先删后建空窗；move 随时
6. **失败隔离**：动作级 try / except + 日志，一个文件失败不拖垮整批
7. **下游无状态**：parser / chunker / embedder 是纯函数，注册表 / 索引存储由编排层统一管

> 本质是状态机：注册表持久化状态，五态是 diff，动作是状态转移，索引操作是副作用。**崩溃后从注册表 + 重新 diff 即可恢复**。

### 2.7 agents_rag 落地

```
扫描 data/raw/ → 流式 hash(+预筛) → 与注册表 diff → 五态 action list
  → 两阶段执行: 先 new/update(建新) → 后 delete(清旧) → move(更路径)
  → 每动作: 多索引原子写入 + 注册表同步 + 操作日志
doc_id = hash(content) + namespace
```

### 2.8 采集层常见陷阱

| 陷阱 | 后果 |
|------|------|
| 用路径 / 文件名作身份 | 一移动就误判，重复索引 |
| 大文件一次性读入算 hash | 爆内存 |
| 用部分 hash 加速 | 漏检变更，脏数据 |
| 只做四态、无「移动」 | 移动文件被误删 + 重建，浪费且有空窗 |
| 多源按文件名去重 | 不同内容误合并 |
| 边检测边执行 | 状态不一致 |
| 先删后建 | 中间空窗丢数据 |
| 删改不靠 chunk_ids | 残留 / 误删 |

### 2.9 异常恢复：崩溃后如何收敛

崩溃最棘手的是**部分写入**（写了向量库没写 BM25、或写了索引没更注册表）。靠「幂等 + 重放 + 校验」恢复，不用 2PC：

```
恢复流程（启动时）:
1. 一致性校验（索引 ↔ 注册表）：孤儿清理、缺失重标
2. 重新 diff（文件 vs 注册表 → 新 action list）
3. 幂等重放（已完成的自动跳过）
4. 继续未完成
```

**关键：写入前清残留**——每个 new/update 在写入前先按 doc_id 清理它的所有残留 chunk，部分写入崩溃后重放会「先清残留再重写」，自动收敛：

```python
def apply_new_or_update(doc):
    delete_chunks_by_doc(doc.doc_id)   # 先清残留（即使旧的不完整）
    chunks = pipeline(doc)
    write_all_indexes(chunks)
    registry.upsert(doc.doc_id, ...)
```

加速：操作日志（WAL）+ checkpoint 定位进度。向量库 / BM25 是异构存储，跨存储事务难做真 2PC，用「幂等 + 重放 + 校验」等价达成最终一致性。

> 本质：action 幂等 + 注册表为真相源 → 恢复 = 校验 + 重新 diff + 重放，崩溃重跑一遍就一致。

### 2.10 update 中间态一致性：新旧并存如何不脏读 ⭐

「先建新后删旧」的窗口期，索引里新旧共存，查询会召回重复。**解法：把「删旧」从物理删除改成「逻辑标记 + 查询过滤」**——旧 chunk 标 `superseded`，查询过滤 `status=active`，物理删除延后：

```
update 流程:
1. 新 chunk 写入        metadata: {version: N+1, status: active}
2. 旧 chunk 标记取代    metadata: {status: superseded}   # 不物理删
3. 查询 where status=active → 只看到新
4. 延迟批量物理删除     superseded 的旧 chunk
```

**chunk 元数据需补 `version` + `status`（active / superseded）两个字段**——既是 update 一致性的基础，也强化崩溃恢复。复用 §7.4 的 metadata 过滤，零额外机制：

```python
collection.query(query_vec, where={"status": "active"}, n_results=k)
```

> 等价方案：注册表记 doc 的 current_version，查询只召回该版本（蓝绿切换）。agents_rag 用 status 方案（chunk 级、复用过滤）。

**核心 takeaway**：采集层的关键是**以内容指纹为身份 + 文档注册表（带 chunk_ids）为真相源 + 五态 diff + 精确动作（先建后删、多索引原子、幂等）+ 异常恢复（幂等重放）+ 中间态隔离（status 过滤）**。这是整条增量管线正确性的根基。

## 3. 文档解析（质量上限的瓶颈）⭐⭐⭐

知识库构建里**最难、最影响上限**的一步。解析不好，后面全是无米之炊。功夫的 80% 花在 PDF 上。

### 3.1 本质：一次「呈现层 → 语义层」的翻译

所有文档格式都是为**给人看 / 给渲染器画**设计的，不是为机器理解。解析的本质，是把「视觉版面」翻译成「结构化语义」：

```
PDF/Office/HTML（呈现层：坐标、样式、标签）
        ↓ 解析
Document（语义层：标题层级、段落、表格、列表 + 位置元数据）
```

不同格式的「语义保留度」天差地别，决定难度：

| 格式 | 语义保留度 | 难度 |
|------|-----------|------|
| Markdown | 高（标题/列表是显式语法） | 极低 |
| HTML | 中高（有标签，但混导航/广告噪音） | 低-中 |
| Office（docx/xlsx/pptx） | 中（有结构标签，但版式干扰） | 中 |
| **PDF** | **低（只有坐标，无语义）** | **高** |
| 扫描件 PDF | 零（只有像素） | 极高（要 OCR） |

PDF 是地狱难度，也是企业文档的大头，所以解析功夫 80% 花在 PDF 上。

### 3.2 PDF 解析深水区

**PDF 内部有什么**：一组**绘制指令**（在坐标 (x,y) 处画字符）。它没有「段落 / 标题 / 表格」概念——这些是人脑补出来的。所以解析是**反向工程**：从带坐标的字，重建人看到的结构。

**三代解析技术的演进**：

| 代 | 代表 | 方法 | 特点 |
|----|------|------|------|
| 第一代 | pdfplumber / PyMuPDF | 规则 + 启发式（坐标聚类成行/段） | 快、可控，但脆弱，复杂版面崩 |
| 第二代 | LayoutParser / deepdoctection | 检测模型找区域 + 规则重组 | 版面识别准，但表格/顺序仍弱 |
| 第三代 | **Docling / MinerU / Nougat** | 端到端文档理解模型（视觉 + 结构联合） | 表格、公式、多栏都强，当前主流 |

> agents_rag 选 Docling/MinerU 即因它们是第三代，能力代差明显。

**PDF 解析要解的 5 个子问题**：

- **a. 版面分析（Layout Analysis）**：把页面切成区域（文本块 / 标题 / 表格 / 图片 / 公式 / 页眉页脚），底层是目标检测模型。决定每块「是什么类型」。
- **b. 阅读顺序还原**：多栏文档，坐标排序会把左右栏混排；要识别「栏」，再按「栏内自上而下、栏间自左而右」排序，跨页还要处理段落承接。
- **c. 表格识别（最值钱）**：两步——**结构识别**（几行几列、合并单元格）+ **内容提取**。差的拆成散字，好的输出结构化（行列矩阵 / Markdown 表）。对企业财务/法规文档决定可用性。
- **d. 公式识别**（学术 PDF）：把公式图片转 LaTeX，MinerU/Nougat 强项。
- **e. OCR（扫描件）**：PDF 无文字层时，先**文字检测**（定位）再**文字识别**（认字）。RapidOCR/PaddleOCR 主流，中文友好，CPU 可跑。

**OCR 触发判断**：解析后若**文本量异常少**（如每页 < N 字符），判定为扫描件，自动转 OCR 路径。

### 3.3 其他格式（相对简单）

- **Office**：结构化程度好。`python-docx` 读 Word（段落 + 标题样式 + 表格），`openpyxl` 读 Excel（直接行列），`python-pptx` 读 PPT。难点是样式 → 语义映射（「加粗大字」算不算标题）。
- **HTML**：`trafilatura` 直接抽正文（去导航 / 广告 / 侧栏），`BeautifulSoup` 兜底清洗。
- **Markdown**：天然结构化，按标题 / 段落解析，最省心。

### 3.4 统一输出模型（关键设计）⭐

不同 parser 输出各异，**必须统一到一个内部 `Document` 结构**，下游（分块 / 索引）才能不关心来源格式：

```
Document
├── doc_id, source, doc_type
├── sections: list[Section]        # 按标题层级组织的章节树
│   └── Section
│       ├── heading, level         # 标题 + 层级（H1/H2…）
│       ├── blocks: list[Block]    # 段落 / 表格 / 列表
│       │   └── Block(type, text, page, bbox, table_data?)
│       └── children: list[Section]
└── metadata
```

设计要点：

- **类型化 Block**（paragraph / table / list）：分块时按类型不同处理（表格不能切断）
- **保留位置**（page、bbox）——引用溯源的根
- **章节树**——结构感知分块的输入

这一层做不好，后面分块会很别扭。

### 3.5 工具选型对比

| 工具 | 强项 | 弱项 |
|------|------|------|
| **Docling** | 输出结构化（标题/段落/表格带类型），表格识别强，输出 Markdown/JSON | 极复杂学术 PDF 偶有遗漏 |
| **MinerU** | 复杂中文/学术 PDF 极强（公式、双栏、表格），保留版面 | 依赖较重（带模型） |
| **Nougat** | 学术 PDF 公式识别强（转 LaTeX / Markdown） | 偏学术、通用文档不如 Docling |
| Unstructured | 全格式通用、生态广 | 表格一般，中文 PDF 不如前两者 |
| PyMuPDF / pdfplumber | 轻量、可控 | 需自己处理结构逻辑 |

### 3.6 降级链与路由（工程关键）⭐

> **演进注记（2026-07）**：当前实现已从「顺序降级」升级为**诊断式降级**——`assess_quality` 产出 `poor_reason`（`scan`/`layout`），router 按原因选对应兜底（`scan`→RapidOCR 强制图像 OCR、`layout`→MinerU 复杂版面），**只降一级**，刻意不做多级串行（Docling→MinerU→OCR）。详见 `openspec/changes/diagnosed-pdf-fallback/`。下方（含 §3.9 代码级）顺序降级链为本节早期教学描述——概念仍适用，工程实现以诊断式为准。

`ParserRouter` 是解析层的调度中心：

```
按扩展名选主 parser
   ↓
解析 → 评估质量（文本量/表格完整度）
   ↓ 质量差
降级到备选（如 PDF→MinerU；文本过少→OCR）
   ↓ 仍失败
记录日志、跳过该文件（不中断批量）
```

原则：**永远要有兜底，单文件失败不能拖垮整批索引**。

### 3.7 解析质量评估

经验信号：每页平均字符数（异常低 → 可能扫描件 / 解析失败）、表格识别比例、标题层级分布、乱码 / 空白占比。可做成解析后自动检查，质量差的文件告警人工复核。

### 3.8 agents_rag 落地

```
file → ParserRouter
        ├─ .pdf  → Docling → (质量差?) → MinerU → (文本少?) → RapidOCR
        ├─ .docx → python-docx
        ├─ .xlsx → openpyxl
        ├─ .pptx → python-pptx
        ├─ .html → trafilatura + BeautifulSoup
        └─ .md   → markdown 解析
        ↓
   统一 Document(sections + blocks + 位置元数据)
```

### 3.9 ParserRouter 降级链的工程实现（代码级）

§3.6 讲了降级链概念，这里到代码级。三个部件：**路由决策 + 质量评估 + 降级循环**。

**质量评估函数（降级的判据）**——解析后算几个快信号：

```python
def assess_quality(doc) -> QualityReport:
    return QualityReport(
        chars_per_page=...,     # 异常低 → 可能扫描件
        table_count=...,        # 有表格标志但无结构化表 → 漏表
        heading_structure=...,  # 标题层级是否合理
        garbage_ratio=...,      # 乱码 / 空白占比
    )

def is_poor(report) -> bool:
    return report.chars_per_page < MIN or report.garbage_ratio > MAX
```

**降级循环（PDF 链为例）**——每级解析后 assess，不达标进下一级：

```
Docling → assess → 差? → MinerU → assess → 差/文本少? → RapidOCR → assess → 仍差? → 记日志跳过
```

**工程结构**：

```python
class ParserRouter:
    def __init__(self, chain: list[Parser]):   # PDF: [Docling, MinerU, OCR]
        self.chain = chain
    def parse(self, file) -> Document | None:
        for parser in self.chain:
            try:
                doc = parser.parse(file)
                if not is_poor(assess_quality(doc)):
                    return doc                # 达标 → 返回
            except ParseError:
                continue                      # 该 parser 失败 → 试下一个
        log_skip(file)                        # 全军覆没 → 跳过
        return None
```

**要点**：chain 按格式配置；每级幂等、可独立失败；评估信号要快（不算 embedding）；全部失败不抛异常，跳过 + 日志（不中断批量）。

### 3.10 表格全链路保真（解析 → 分块 → 索引）

表格是领域文档命门，全链路不能崩。四层都要护住：

| 层 | 做法 |
|----|------|
| **解析** | Docling / MinerU 输出结构化表（行列矩阵 / Markdown 表），存为 typed block（`type=table`，带 `table_data`） |
| **分块** | 表格作**原子块**，绝不内部切断；大表策略——整体保留（优先保真）或按行切但每块**重复表头** |
| **索引** | 表格 embedding 用「表头 + 若干行」文本表示 / Markdown 渲染；元数据标 `block_type=table` |
| **回传** | 父子分块：表格整体作父块，按行 / 语义切子块检索，命中回传**整表** |

> 关键：表格的「类型化」从解析层一路透传到回传层——`type=table` 让分块豁免、让索引特殊处理、让回传渲染整表。

### 3.11 跨页表格合并

一个表跨多页，解析器常把每页的「表头 + 部分行」当独立表。

**检测**：连续页的表格 block，且**列数相同**、上方无新标题 → 多半是同一张表的续表。优先用原生支持跨页合并的解析器（Docling / MinerU 较新版本支持）。

**合并**：拼接行，**去重复表头**（第二页起的表头丢弃）；`page` 取多页范围（如 `page=[3,4]`），溯源仍准。

**后处理兜底**（解析器不支持时）：

```python
def merge_adjacent_tables(blocks) -> list[Block]:
    # 相邻 table block 列数一致 → 合并，去重复表头
    ...
```

合并后仍是 typed table block，走 §3.10 的全链路保真。

### 3.12 解析层常见陷阱

| 陷阱 | 后果 |
|------|------|
| 用第一代规则解析器啃复杂 PDF | 表格 / 多栏全崩 |
| 不做 OCR 兜底 | 扫描件直接丢字 |
| parser 输出不统一 | 下游分块要为每种格式写特例 |
| 不保留 page/bbox | 无法引用溯源 |
| 表格被当普通文本切 | 表格数据损毁 |
| 单文件失败中断整批 | 索引管线脆弱 |

**核心 takeaway**：解析是「呈现 → 语义」的翻译，PDF 是主战场，靠**第三代文档理解模型 + 统一输出模型 + 降级链兜底**。

## 4. 清洗归一化（净化层）⭐

解析之后、分块之前。把解析输出的「带噪原始内容」整理成**干净、规范、适合分块与 embedding** 的形式。它是「承上（解析）启下（分块）」的净化层。

「轻」不等于「可省」——脏数据进去，后面 chunk 和 embedding 全被污染。

### 4.1 为什么需要

解析输出的内容天然带噪：

- **页眉页脚每页重复**（「第 X 页 | 公司名 | 章节名」）——同一噪声被切成几十个 chunk
- **水印、批注、修订标记**
- **PDF 硬换行**导致碎句（一句话被切成几行）
- 多余空白、空行
- 编码 / 乱码、特殊字符
- 全半角标点混用
- OCR 错字、孤立字符

不清这些噪声，会**稀释 embedding 信号、制造无效 chunk、干扰检索排序**。

### 4.2 清洗的六类操作

| 类别 | 做什么 |
|------|--------|
| **去噪** | 页眉页脚、页码、水印、目录、脚注引用标记 |
| **空白规整** | 合并 PDF 断行、去多余空格 / 空行 |
| **编码归一** | 统一 UTF-8、修常见乱码 |
| **标点归一** | 全半角统一、中文标点规范 |
| **结构清洗** | 去空标题、合并碎片段 |
| **OCR 后处理**（可选） | 规则或小模型纠正常见错字 |

### 4.3 关键原则：克制度 ⭐

清洗最忌「用力过猛」：

- **去噪 ≠ 误删正文**：表格里的编号、公式符号、代码缩进不是噪声，删了就丢信息
- **只动文本层，不破坏结构**：解析得到的标题层级、表格、列表要保留
- **保留位置元数据**（page / bbox）——溯源根基不能在清洗时丢
- **可追溯**：记录清洗前后的差异，便于回溯调试

### 4.4 边界：和解析、分块怎么分工

- **依赖解析**：去页眉页脚其实可以在解析阶段用版面分析直接做掉（识别出「页眉区」丢弃）。清洗与解析的边界是灵活的。
- **服务分块**：清洗的验收标准是——「分块拿到的是干净的」。
- 实践中：**版面级去噪归解析，文本级规整归清洗**。

### 4.5 常见陷阱

| 陷阱 | 后果 |
|------|------|
| 过度清洗 | 删掉有用信息（编号、符号、缩进） |
| 一刀切规则 | 不同文档类型需要不同清洗规则 |
| 不留原文 | 出问题无法回溯 |
| 清洗时丢元数据 | 引用溯源断链 |

### 4.6 agents_rag 落地

- 作为 pipeline 的独立步骤（解析后、分块前），或解析的后处理
- 清洗规则 **按 doc_type 可配置**（PDF / HTML / Office 规则不同）
- 保留 page / heading 等元数据
- 记录清洗日志，可追溯

### 4.7 实体规范化（同一实体的多种写法）

同一实体多种写法（「IBM」vs「国际商用机器」vs「IBM Corp.」）导致召回不全：查 IBM 找不到写「国际商用机器」的文档（BM25 全漏、向量部分漏）。

**方案谱（轻 → 重）**：同义词表扩展（检索时扩 query，轻）/ 实体归一化（清洗时替换，中）/ NER + 实体链接（重，近 GraphRAG）。

**关键权衡：清洗时改 vs 检索时扩** ⭐——清洗时替换会**改原文**（溯源困惑）且要跑 NER（成本高）；检索时扩展**不改原文**、更安全。

> agents_rag 倾向**检索时扩展**（同义词 + 查询改写），清洗阶段只做轻量归一化（全半角、大小写）。实体规范化本质是检索质量问题，放查询侧更合适。

### 4.8 PII 敏感信息脱敏（合规）⭐

企业文档含 PII（身份证、手机、邮箱、姓名、地址、银行卡），直接入库违反隐私合规。更糟：**PII 一旦进索引，会被检索 + 生成泄露**（用户问「张三的联系方式」，模型从 chunk 直接吐出）。**入库前（清洗阶段）是唯一可靠防线**。

**识别**：规则（正则，手机 / 身份证 / 邮箱 / 银行卡 / IP）+ NER（人名 / 地址 / 组织）。

**脱敏策略**：

| 策略 | 例子 | 适用 |
|------|------|------|
| 掩码 | `138****1234` | 保留格式 |
| 泛化替换 | → `[PHONE]` / `[ID]` | 完全去标识 |
| 删除 | 去掉 | 不需要时 |
| 假名化 | 张三 → 用户A | 可逆（映射加密另存） |

**权衡**：脱敏过度丢关键信息（人名是查询主体）；不足则泄露。假名化可逆（审计可还原），掩码 / 泛化不可逆。

**落地**：清洗阶段正则 + NER 识别 → 按策略脱敏；脱敏后入库，原始文本加密另存；策略可按 ACL 分级；记录脱敏日志可审计。

**核心 takeaway**：清洗是克制地去噪——**不动结构、保留元数据、可追溯**；进阶上，**实体规范化倾向检索侧（同义词扩展）、PII 脱敏必须在清洗侧（合规防线）**。

## 5. 分块（Chunking）⭐⭐⭐ —— 知识库构建的核心技术

如果说解析是「拿出来」，分块就是「切得对」。**这是知识库质量最关键的杠杆**。

### 5.1 本质再认识：chunk 是「检索单元」不是「阅读单元」

最关键的认知转变。很多人按「给人读的段落」来切 chunk，是错的。chunk 的设计目标是**检索效率**：让相关语义能被精准命中。它不需要自带完整上下文（上下文可靠父块 / 窗口回传补足）。这个认知一旦建立，就理解了为什么子块可以切得很碎、chunk 可以不带完整上下文、父子分块与句子窗口这些「分离粒度」设计才合理。

### 5.2 为什么要分块（核心矛盾）

- **检索粒度**：整篇文档做向量，召回时信号被稀释（一个向量代表几万字，必然模糊）
- **上下文预算**：LLM 输入有限，要精准喂相关片段
- **召回精度**：块越小越精确，但太小会丢上下文 → **核心矛盾**

> 经验法则：chunk 太大 → 噪声多、token 浪费、召回稀释；chunk 太小 → 语义残缺、答案缺上下文。**没有银弹，要按数据调**。

### 5.3 分块策略（从简单到高级）

**① 固定大小分块**：按 token / 字符数硬切 + overlap。最简单，但会从句子/表格中间切断，基线可用、质量差。

**② 递归字符分块**：按分隔符优先级切（`\n\n` 段落 → `\n` → `。` 句 → 空格）。主流基线，比固定好，但仍忽略文档结构。

**③ 结构感知分块**（推荐）：利用解析得到的标题层级，按「章节 / 小节 / 段落」自然边界切。一块属于一个语义单元，质量最高。

**④ 语义分块**：用 embedding 相邻句相似度找「语义断点」。自适应，但要预计算 embedding，慢、成本高。

**⑤ 格式感知**：表格、列表、代码块必须整体保留，绝不在内部切断。常被忽略，但对表格密集的领域文档（财务、法规）是致命的。

### 5.4 结构感知分块：算法实现

输入是解析得到的章节树（§3.4 的 `Document → sections → blocks`）：

```
遍历章节树:
  对每个 section:
    收集其下 blocks（段落/表格/列表）
    累加 block 文本:
      if 累加长度 ≤ chunk_size:   继续聚合
      else:                        切出一个 chunk，开启新 chunk
    特殊 block（表格/代码/列表）:  整体作为一个 chunk，不参与累加切断
  每个 chunk 继承: section_path（章节路径）, page, heading
```

三个要点：

- **尊重 block 边界**：切断只发生在 block 之间，绝不切断一个段落 / 表格
- **标题继承**：chunk 带 `section_path`（如「第三章 > 3.2 分块 > 算法」），检索命中后提供「这个片段在讲什么」的上下文
- **特殊结构豁免**：表格 / 代码 / 列表作为原子块，再大也整体保留

### 5.5 父子分块：工程实现 ⭐

**数据结构**：

```
子块 child_chunk:  id, parent_id, text(小), embedding, page, heading
父块 parent_chunk: id, text(大, 含完整段落/小节), page, heading
```

**存储（双索引）**：向量索引只建在**子块**上（子块负责被检索）；父块单独存一个 KV（parent_id → 父块文本），不建向量索引。

**查询流程**：

```
query → 向量检索子块 top_k → 得到命中的 child_ids
      → 取 parent_ids（去重）
      → 查 KV 取父块文本
      → 父块回传给 LLM
```

**进阶：AutoMerging（自动合并）**：当一个父块下的**多个子块都被命中**，说明这一整段高度相关——直接回传整个父块，而非返回多个重叠片段。避免回传内容冗余和截断。LlamaIndex 的 `HierarchicalNodeParser` 即此思路。

**变体：句子窗口**：不显式建父子，而是检索单句、回传时用「该句 ± N 句」拼出窗口。实现更轻，但上下文控制不如父子灵活。

> **agents_rag 选型**：父子分块（结构感知切父块 → 父块再切子块 → 子块建索引）。比句子窗口更适合有清晰结构的领域文档。

### 5.6 调参方法论（最重要、最被忽视）⭐

**核心原则：chunk_size 不是拍脑袋定的，是评测出来的。** 概览给的「中文 300–600 字」是范围，不是答案。调参流程：

1. 建小评测集（30–50 条 query + 每条「应被召回的文档」）
2. 选一组候选 chunk_size（如 256 / 384 / 512 / 768），相同 overlap 比例
3. 每组建索引，跑评测集，记 **recall@k** 和**答案质量**
4. 选 recall 最高且 chunk 语义完整的那个

**联动关系（关键）**：

- **chunk_size ↔ top_k**：块越小，单个块信息越少，要多召回几个（top_k 大）；块越大，少召回几个。两者一起调。
- **chunk_size ↔ overlap**：overlap 防边界丢失，但过大则存储 / 计算冗余。一般 10–20%。
- **chunk_size ↔ embedding 上下文**：embedding-3 单条上限 3072 token，块别超（远不会超）。

**经验锚点**（起点，非终点）：

- 事实 / 定义密集（法规、手册）：偏小（300–400 字），精准
- 叙述 / 论述密集（报告、论文）：偏大（500–700 字），保上下文

### 5.7 特殊结构处理（领域文档的命门）

| 结构 | 处理 | 原因 |
|------|------|------|
| **表格** | 整体一个 chunk，转 Markdown 表 | 切断 = 行列关系崩 |
| **列表** | 整体保留 | 切断 = 条目脱离语境 |
| **代码块** | 整体保留，保缩进 | 切断 = 语法 / 逻辑断 |
| **公式** | 整体保留 | 切断 = 不可读 |

表格密集的领域（财务报表、产品规格），这步做不好整张表变乱码，前面解析再好也白搭。

### 5.8 进阶：层级索引（多粒度）

同一文档切**多个粒度**（如 256 / 512 / 1024 三层），建多套索引。查询时：简单事实查询 → 小粒度（精准）；综述类查询 → 大粒度（完整上下文）；用「查询路由器」判断走哪层。成本高（多套索引），适合查询类型多样的场景。agents_rag 第一版用单层父子即可，这是后续优化方向。

### 5.9 中文分块细节

- **不按裸字符数切**：会切断词（如「机器」切成「机」/「器」）
- **按标点 / 段落优先级切**：`。！？\n\n` → `\n` → 逗号
- **token vs 字符**：embedding 按 token 计费，中文 1 字 ≈ 1–2 token；chunk_size 设定时注意是「字符」还是「token」口径
- **jieba 的作用**：主要在 BM25 侧；分块侧用标点边界足够

### 5.10 分块质量评估指标

几个可量化信号：

- **平均 chunk 长度分布**：是否集中在预期区间，有无大量极短 / 极长块
- **截断率**：在句子 / 表格内部被切断的 chunk 占比（应趋近 0）
- **语义完整度抽检**：人工看一批，是否语义自洽
- **召回测试**：recall@k 是最终评判

### 5.11 元数据附着（每个 chunk 必带）

```
doc_id | source | page | heading | chunk_idx | parent_id | section_path
```

**没有元数据 = 无法引用溯源 = 企业场景不可用。**

### 5.12 agents_rag 落地

```
Document(章节树)
   → 结构感知切父块 (尊重 block 边界, 表格/代码豁免)
   → 父块再切子块 (子块粒度由 chunk_size 控制)
   → 子块带 parent_id + section_path + page + heading
   → 子块建向量索引; 父块存 KV
   (查询: 子块命中 → AutoMerging 取父块回传)
```

参数（chunk_size / overlap / top_k）全部配置化，配评测集调优。

### 5.13 分块层常见陷阱

| 陷阱 | 后果 |
|------|------|
| 照抄默认 chunk_size | 不适配你的数据 |
| 只用固定大小分块 | 切坏句子 / 表格 |
| 表格 / 代码被切断 | 结构数据损毁 |
| 检索粒度=回传粒度（无父子） | 要么召回稀释，要么上下文不全 |
| 不调参、不评测 | 质量靠玄学 |
| overlap 过大 | 冗余、成本高 |
| chunk 不带 section_path | 命中后缺上下文定位 |

### 5.14 父子分块代码级实现

§5.5 讲了概念与流程，这里到可运行实现，并补上概念版没展开的细节。

**数据结构（具体化）**：

```python
@dataclass(frozen=True)
class ParentChunk:
    id: str; doc_id: str; text: str
    page: int | tuple; heading: str; section_path: str

@dataclass(frozen=True)
class ChildChunk:
    id: str; parent_id: str; doc_id: str; text: str
    page: int; heading: str; section_path: str
    version: int; status: str          # active / superseded（§2.10）
    # embedding 只在向量库，不存对象
```

子块带 `parent_id` + `version` + `status`（呼应 §2.10 update 一致性）。

**切分算法（父 → 子）**：

```python
def chunk_document(doc, parent_size, child_size, overlap):
    parents = structural_split(doc, parent_size)              # 结构感知切父块
    children = []
    for parent in parents:
        for i, sub in enumerate(sliding_split(parent.text, child_size, overlap)):
            children.append(ChildChunk(
                id=f"{parent.id}__{i}",                       # id 编码 parent_id
                parent_id=parent.id, text=sub, ...,
                version=1, status="active"))
    return parents, children
```

**存储**：父块入 KV（不建向量索引）；子块入向量库 + BM25（metadata 带 `parent_id / version / status`）。

**查询回传 + AutoMerging（代码级）**：

```python
def retrieve(query, top_k=20, merge_threshold=2):
    child_hits = vectorstore.query(embed(query), where={"status": "active"}, n_results=top_k)
    groups = group_by(child_hits, key=lambda c: c.metadata["parent_id"])
    results = []
    for parent_id, hits in groups.items():
        if len(hits) >= merge_threshold:
            results.append(parent_store.get(parent_id))       # 回传整个父块
        else:
            results.extend(hits)                              # 命中少 → 回传子块
    return dedupe_and_rank(results)
```

AutoMerging：按 `parent_id` 聚合，命中多的父块整体回传；`merge_threshold` 调激进度。

**句子窗口变体（轻量替代）**：检索 sentence chunk，回传取句 ± N，无需父块存储，但窗口固定、不能按结构切。

**增删改的父子协同**（呼应 §2，概念版未展开）⭐：

| 操作 | 父子协同 |
|------|---------|
| 新增 | 切父 → 切子，父入 KV、子入向量库 + BM25 |
| 更新 | 新父子写入 → 旧父子**整体标 superseded** → 延迟物理删 |
| 删除 | 按 doc_id 同时清父（KV）+ 子（向量库 + BM25） |

> update 的 superseded 标记要**父子一起标**——父子是原子单元，只标一边会不一致。

### 5.15 父块大小的界定

子块有 `chunk_size`（检索粒度），父块大小没有标准答案，但要平衡**上下文完整 ↔ token 预算 ↔ 相关性**。父块是「回传单元」不是「检索单元」。

**三种界定方式**：

| 方式 | 做法 | 缺点 |
|------|------|------|
| ① 结构单元 | 父块 = section / 段落（章节树） | 大小不均 |
| ② 固定大小 | 父块 = `parent_chunk_size` | 可能切断 section |
| ③ 倍数关系 | 父块 = `chunk_size × N` | 仍固定大小 |

**推荐：结构感知优先 + 上下限约束** ⭐

```
父块切分:
  1. 按章节树切 section（语义边界优先）
  2. if section > parent_max_size:  再切（按段落）
  3. if section < parent_min_size:  合并相邻 section
```

**最终约束：token 预算反推**——父块硬上限来自 LLM 上下文：

```
父块平均大小 ≤ 检索上下文预算 / 回传父块数
（例：预算 ~8000 token、回传 5 父块 → 父块 ≤ ~1600 token）
```

**父子联动**：父块 ≈ 子块 3–5 倍；父块越大 → 子块越多 → AutoMerging（§5.14）越易触发，需配合调 `merge_threshold`。

**多层父子（进阶）**：祖父 → 父 → 子多层（LlamaIndex `HierarchicalNodeParser`），第一版不做。

**agents_rag 落地**：父块 = section，`parent_max_size` 由 token 预算反推（~1800 字），超切小合；子块 ~400 字；父子比 ≈ 4–5。

**核心 takeaway**：分块的本质是把文档切成「最佳检索单元」——靠**结构感知切分 + 父子分离粒度 + 评测驱动调参 + 特殊结构豁免**四件套。其中**调参方法论是最该建立的能力**：别抄参数，用评测集调。父块大小则靠**结构语义边界 + token 预算反推的 parent_max_size** 界定。

## 6. 向量化与索引（最标准的一步）⭐⭐

向量化是「语义召回」，BM25 是「精确词召回」，两者互补才是完整召回。这一步原理标准化，但工程化（并发 / 缓存 / 双索引同步）有讲究。

### 6.1 Embedding 原理：为什么向量能做语义检索

文本经 embedding 模型映射成**高维向量**（embedding-3 是 2048 维）。关键性质：**语义相近的文本，向量在空间中距离近**。

```
"如何重置密码"      ──→  [0.12, -0.33, ...]  ┐
"忘记密码怎么办"    ──→  [0.11, -0.31, ...]  ├─ 距离近（语义同）
"密码重置流程"      ──→  [0.13, -0.35, ...]  ┘
"今天天气不错"      ──→  [0.85, 0.02, ...]   ── 距离远（语义不同）
```

这是关键词检索做不到的——「重置密码」和「忘记密码」没有共同关键词，但语义一样，向量能匹配。

**bi-encoder 架构**：query 和 doc **独立编码**成向量。doc 可离线编码建库，查询时只编码 query 一次，再和库里所有 doc 向量算距离。这是向量检索快的基础（也是精度上限的来源——查询侧 rerank 的 cross-encoder 才互补）。

### 6.2 相似度度量

| 度量 | 说明 | RAG 用法 |
|------|------|---------|
| **余弦相似度** | 比向量方向（归一化后），忽略长度 | **默认首选** |
| 点积 | 未归一化的余弦，向量长度影响结果 | 归一化向量下等价余弦 |
| 欧氏距离 | 几何距离 | 少用 |

为什么默认余弦：文本 embedding 的「语义强度」不应被向量长度干扰，余弦只看方向（语义朝向），最稳。

### 6.3 向量索引算法（ANN）⭐ —— 工程核心

库一大，暴力比对所有向量（Flat）就慢了。于是有**近似最近邻（ANN）**算法，用一点精度换巨大速度提升。

| 算法 | 原理 | 特点 |
|------|------|------|
| **Flat（暴力）** | 逐一算距离 | 精确，但 O(N)，仅小库可用 |
| **HNSW** | 分层小世界图，导航式逼近 | 查询快、召回高，**主流**；内存高、构建慢 |
| **IVF** | 先 K-means 聚类，查询只扫几个簇 | 可调探测数（nprobe）平衡速度/召回 |
| **PQ（乘积量化）** | 向量压缩省内存 | 省内存但损精度，常和 IVF 组合（IVF-PQ） |

**关键权衡三角**：召回率 ↔ 查询速度 ↔ 内存占用，没有全占。

> agents_rag 用 Chroma，默认 HNSW——中小规模 + 本地优先场景的最佳选择。规模化时 HNSW 的内存开销才需关注。

### 6.4 Embedding 批处理与并发（工程化）⭐

embedding 是付费 API 调用，工程化要点：

- **批处理**：单次最多 64 条（embedding-3 限制）、单条 ≤ 3072 tokens，批量摊薄网络开销、降本提速
- **维度权衡**：2048 最高精度（领域文档推荐）/ 1024 平衡 / 512 省
- **并发控制**：信号量限在途请求数（受账号等级限流：V0=50 … V3=500），超了 429
- **重试**：tenacity 指数退避，区分可重试（429/5xx）与不可重试（鉴权错）
- **进度持久化**：批量时记录「已处理到哪」，中断可续跑（别白花钱重算）
- **异步**：aiohttp / asyncio 并发请求，别串行等

### 6.5 Embedding 缓存策略 ⭐

**最被低估的省钱/提速手段**。同一块文本（如未变更 chunk）不应重复 embed。

- **缓存键**：文本的 hash（SHA-256），文本不变则向量不变
- **存储**：本地 KV（sqlite / 磁盘文件）
- **复用场景**：增量索引时未变更文档的 chunk 命中缓存；调参重跑时相同 chunk 复用
- **收益**：大幅降本 + 实验可复现（同样输入同样向量）

> 没有缓存的 RAG，每次调参重索引都全量付费重 embed——成本和迭代速度双重打击。

### 6.6 BM25 原理与中文适配 ⭐

**为什么需要它**：向量对「型号、编号、人名、专有术语」召回差（这些词语义向量不够区分）。BM25 是经典全文检索算法，专攻精确词匹配。

**BM25 是什么**：TF-IDF 的概率论改进版。直觉：

```
score(query, doc) = Σ 对每个 query 词:
    IDF(词) × TF饱和(词,doc) × 文档长度归一化
```

- **IDF**：词在全局越稀有权重越高（「的」低，「GLM-4.5」高）
- **TF 饱和**：词频贡献有上限（10 次不比 5 次好两倍）
- **文档长度归一化**：长文档不因「字多」占便宜

**中文适配（关键）**：BM25 建立在「词」上，中文无空格分词，**必须先分词**（jieba），否则按单字 BM25 效果差、噪声大。

```
原文：智谱GLM4.5模型
jieba：智谱 / GLM / 4.5 / 模型  →  BM25 对「词」建索引
```

### 6.7 双索引的存储与一致性 ⭐

向量索引和 BM25 索引是两套独立存储，**必须保持一致**：

- **ID 对齐**：两边用同一个 `chunk_id`，查询结果能对应回同一块
- **同步增删改**：文档变更时，向量库 upsert + BM25 重建/增量，不能只更一边
- **rank_bm25 持久化**：默认内存型，要自己 pickle 到磁盘（或用 bm25s 支持磁盘）

### 6.8 向量库选型与迁移路径

| 向量库 | 定位 | 适用 |
|--------|------|------|
| **Chroma** | 嵌入式、零运维 | 起步 / 中小规模 / 本地（agents_rag 起步选） |
| **Qdrant** | 高性能、强过滤、可服务化 | 中大规模、生产 |
| **Milvus** | 分布式 | 大规模、海量 |
| **FAISS** | 库（非服务） | 要自己包一层服务 |

**迁移路径（先小后大的关键）**：抽象 `VectorStore` 接口（upsert/delete/query），初期 Chroma，规模化时换 Qdrant——业务代码不动。

### 6.9 质量评估

- **embedding 健康度**：相关对 vs 不相关对的相似度分布是否分离（分得开才好用）
- **ANN 召回率**：ANN 结果 vs 暴力检索 ground truth 的重叠率（recall@k）
- **向量 vs BM25 召回对比**：看两类各自召回什么，验证互补性

### 6.10 agents_rag 落地

```
子块文本 → (查缓存) → 批量 embedding-3 (并发限流 + 重试) → 写缓存
        → Chroma 向量库 (HNSW) [VectorStore 抽象]
        → BM25 索引 (jieba + rank_bm25, 持久化)
两索引以 chunk_id 对齐, 增删改同步
```

### 6.11 向量化与索引层常见陷阱

| 陷阱 | 后果 |
|------|------|
| 只建向量索引，无 BM25 | 专有名词 / 编号召回漏 |
| 无 embedding 缓存 | 调参重跑全量付费、不可复现 |
| 并发不限流 | 频繁 429、批量中断 |
| 中文 BM25 不分词 | 按字匹配、噪声大 |
| 双索引不同步 | 召回结果对不上 |
| 向量库未抽象接口 | 规模化时迁移痛 |
| 忽略 ANN 召回率 | 看似快，实际漏召回 |

### 6.12 稀疏向量与 ColBERT（BM25 之外的精确召回进阶）

§6.6 的 BM25 是词频统计的词袋（无语义、依赖分词）。两个进阶方案补其不足：

- **稀疏向量（SPLADE）**：文档表示为稀疏高维词向量（少数词有权重），权重由**模型学**（非统计），还能学**术语扩展**。比 BM25 强，仍稀疏可高效检索。
- **ColBERT（late interaction）**：query 和 doc 都编码成 **token 级向量**，检索时 token 级延迟交互。介于 bi-encoder（快但粗）和 cross-encoder（准但慢）之间；代价是存储大（每文档多向量）。

**召回方案谱**：

| 方案 | 匹配 | 精度 | 速度 | 存储 |
|------|------|------|------|------|
| BM25 | 词频统计 | 中低 | 极快 | 小 |
| 稀疏向量（SPLADE） | 学习词权重 + 扩展 | 中 | 快 | 中 |
| bi-encoder（embedding-3） | 语义向量 | 中高 | 极快 | 小 |
| ColBERT | token 级延迟交互 | 高 | 中 | 大 |
| cross-encoder（rerank） | 全交互 | 最高 | 慢 | — |

> agents_rag 第一版用 bi-encoder + BM25；稀疏向量 / ColBERT 是后续优化——BM25 不足又不想上 rerank 时选 SPLADE，要极致精度时考虑 ColBERT。

### 6.13 索引漂移与模型版本管理 ⭐

embedding 模型升级（embedding-3 → 4），新旧向量空间不同，**旧向量和新向量不能直接比**，混用 → 检索错乱。

**漂移触发**：模型升级（语义空间 / 维度变）/ 维度配置变 / 分块策略变。

**处理：模型版本与索引绑定**——chunk 向量带 `embedding_model + embedding_dim` 元数据；索引按模型版本分 collection，不混用；升级 = 新 collection 全量 re-embed → 切换 → 下线旧（**蓝绿**，机制同 §2.10）。

**全量重建协调**：embedding 模型升级**必须全量重建**（每 chunk 向量都重算），不能增量；策略是灰度验证 → 全量重建 → 切换。

**⚠️ 缓存键必须版本化（隐蔽 bug 高发点）**——§6.5 的缓存键不能只是文本 hash：

```python
cache_key = hash(text) + model + dim      # 不是 hash(text)
```

否则换模型后命中**旧模型缓存**，向量错误（极隐蔽）。

**agents_rag 落地**：chunk 元数据带 `embedding_model / dim`；缓存键含 `model + dim`；模型升级走蓝绿全量重建；与 §8 version、§2.10 切换协调。

**核心 takeaway**：向量化是「语义召回」，BM25 是「精确词召回」，**两者互补才是完整召回**；工程上靠**批处理 + 并发限流 + 缓存 + 双索引同步**把这一层做稳。索引算法选 HNSW（Chroma 默认），向量库抽象接口为规模化留路。进阶上，**稀疏向量 / ColBERT 是精确召回的梯度选择、索引漂移靠模型版本绑定 + 缓存键版本化防治**。

## 7. 元数据与（可选）知识图谱 ⭐

元数据看似轻，但它是把「检索命中」转化为「可溯源、可过滤、可解释」的关键。前面分块、向量化多次提到，这里系统讲透。

### 7.1 元数据的角色：贯穿全流程的「粘合剂」

元数据不是某一环节的产物，而是**贯穿解析→分块→索引→检索→生成**的线索。每个 chunk 携带的元数据，决定了能不能：回答时说清来源、按条件过滤、控制权限、去重版本化。

一句话：**没有元数据的 chunk 是「漂浮的文本」，有了元数据才是「可管理、可溯源的知识单元」**。

### 7.2 元数据 schema 设计（分层）⭐

设计成两层，职责清晰：

**文档级（doc-level）**——一个文档一份：

```
doc_id(指纹) | source | doc_type | title | author
created_at | updated_at | version | acl(权限)
```

**块级（chunk-level）**——每个 chunk 一份：

```
chunk_id | doc_id | page | heading | section_path
parent_id | chunk_idx | char_span(字符区间)
```

设计原则：

- **检索 / 溯源要用的字段都进元数据**（page、heading、section_path）
- **可过滤字段单独标**（时间、来源、doc_type、权限）——这些是 pre-filter 的依据
- 用向量库的 metadata 字段承载（Chroma 每个 chunk 可挂一个 metadata dict）

### 7.3 元数据的四大用途 ⭐

| 用途 | 怎么用 | 依赖字段 |
|------|--------|---------|
| **引用溯源** | 回答回溯到「文档名 + 页码 + 片段」 | source, page, char_span |
| **过滤检索** | 按来源 / 时间 / 类型缩小范围 | source, created_at, doc_type |
| **权限控制** | 用户只能检索有权限的文档 | acl |
| **去重 / 版本** | 指纹去重、版本区分 | doc_id, version |

企业场景里，**权限控制**（ACL）尤其关键——不同部门 / 角色只能查到各自权限内的文档，必须靠元数据在**检索层强制过滤**，不能靠 prompt 约束（不可靠）。

### 7.4 元数据过滤检索：pre vs post filtering ⭐

| 方式 | 流程 | 特点 |
|------|------|------|
| **pre-filtering** | 先按元数据过滤候选集 → 再向量检索 | 省算力、结果准；要向量库支持 |
| **post-filtering** | 先向量检索 top_k → 再按元数据过滤 | 通用；但可能过滤后所剩无几 |

- Chroma 支持 `where` 条件过滤（pre-filter 风格），可在向量检索同时施加元数据约束
- **权衡**：pre 省且准（候选集小、不浪费 top_k 配额），但需向量库原生支持；post 通用但 top_k 可能被过滤光（尤其权限过滤后候选骤减）
- **实践**：权限这种「强约束」用 pre-filter；时间范围这种「软筛选」可 post

### 7.5 元数据存储与一致性

- **主存储**：向量库的 metadata 字段（Chroma：每个 chunk 一个 metadata dict）
- **复杂查询**：若要 SQL 式多条件查询，可另存 sqlite 做元数据索引（向量库负责检索、sqlite 负责结构化查询）
- **一致性**：元数据和向量**同生命周期**——增删改 chunk 时，向量与 metadata 同步更新（与 §6.7 双索引一致性一脉相承）

### 7.6 知识图谱 / GraphRAG（高级，可选）⭐

传统 RAG 是**扁平检索**：chunk 之间无关系，各检各的。这对「需要跨文档、多跳推理」的问题力不从心（如「A 公司的供应商的主要客户是谁」要跨多份文档串联）。

**GraphRAG 思路**：

1. 用 LLM / NER 从文档**抽取实体与关系**（实体如「GLM-4.5」、关系如「属于→智谱」）
2. 构建**知识图谱**（节点=实体，边=关系）
3. 查询时结合图遍历 + 向量检索，支持**多跳推理**

**适用场景**：医疗（症状-疾病-药物）、法律（法条-案例-判决）、金融（公司-持股-关联交易）等关系密集领域。

**代价**：实体抽取（LLM 调用，成本高）、图谱构建与维护、查询复杂度上升。微软的 GraphRAG 是代表方案。

> **agents_rag 定位**：第一版**不做**知识图谱（用扁平的父子分块 + 混合检索足够），作为后续进阶方向——当发现「跨文档推理」类问题答不好时再引入。

### 7.7 agents_rag 落地

```
chunk 写入向量库时附带 metadata:
  {doc_id, source, page, heading, section_path, parent_id, chunk_idx, char_span, doc_type, created_at, acl}
检索时:
  where 条件 (pre-filter): doc_type/时间/acl 约束
  命中后: 用 source+page+char_span 生成引用
权限: acl 在 pre-filter 强制, 不进 prompt
```

### 7.8 元数据层常见陷阱

| 陷阱 | 后果 |
|------|------|
| chunk 不带 page/section_path | 无法引用溯源 |
| 权限靠 prompt 约束 | 不可靠、可被绕过 |
| 不过滤就全库检索 | 噪声多、top_k 浪费 |
| 元数据与向量不同步 | 溯源错位、过滤失效 |
| 元数据字段散乱无 schema | 难维护、难查询 |
| 一上来就上 GraphRAG | 过度工程、成本高 |

### 7.9 元数据自动抽取质量

文档级元数据（作者 / 时间 / 标题 / 来源）多靠自动抽取，但常不准（文件属性 `author="admin"`、标题启发式错）。基于错误元数据的过滤 / 溯源会出错。

**抽取来源（多源，质量不一）**：文件系统属性 / 文件内嵌 metadata（常缺失错）/ 内容启发式 / **LLM 抽取**（最准但贵）。

**质量保障**：

- **多源交叉验证**：不一致 → 标记低置信
- **置信度标注**：每字段带 `confidence`，低置信过滤降权或人工复核
- **关键字段校验**：标题等缺失告警
- **允许人工修正**：覆盖自动抽取

**落地**：抽取管线（文件属性 → 内容启发式 → LLM 按需），每字段带 `source + confidence`，低置信不参与强过滤。

### 7.10 复杂权限模型（ACL 进阶）

§7.3 的 ACL 基础是用户列表，企业权限常更复杂：角色继承 / 文档级 + 字段级 / 动态权限（项目 / 时间）/ 否决优先。

**权限模型**：RBAC（角色，简单常用）/ ABAC（属性规则，灵活但复杂）/ 混合。

**在 RAG 检索的落地**：

- chunk 的 acl 要能表达角色 / 规则，不能只是用户列表
- **简化**：索引时把「可见角色集合」展开成 acl 列表存元数据；检索时 `user → 查其所有角色 → pre-filter`
- **复杂**：权限引擎（如 OPA）实时判断可见性（灵活但慢）
- **字段级权限**：极复杂，第一版不做

| 方案 | 特点 |
|------|------|
| 简单（acl 列表 + pre-filter） | 快，不支持动态 / 复杂规则 |
| 复杂（权限引擎实时） | 灵活，但每次检索调引擎，慢 |

> agents_rag 第一版：文档级 RBAC（acl 角色列表 + pre-filter）；字段级 / 动态后续。

**核心 takeaway**：元数据是 chunk 的「身份与上下文」——靠**分层 schema + 四大用途（溯源 / 过滤 / 权限 / 版本）+ pre-filter 强制**把它用起来。权限必须在检索层用元数据强制，绝不靠 prompt。知识图谱是进阶方向，第一版不做。进阶上，**抽取要多源交叉验证 + 置信度、权限靠 RBAC/ABAC 展开成 acl 列表**。

## 8. 持久化与增量维护（运维关键）⭐⭐

构建管线的收尾环节，运维向。决定知识库「能不能长期稳定运行、数据变了能不能低成本更新」。

### 8.1 为什么需要：全量重建的代价

文档不是静态的——会增、删、改。每次变更都**全量重建**索引的问题：

- **慢**：几千份文档重新解析 / 分块 / embedding 要几小时
- **贵**：embedding 重新付费（没缓存的话）
- **中断**：重建期间服务不可用或用旧索引

所以必须**增量**：只处理变化的部分。

### 8.2 变更检测：文档指纹 diff ⭐

核心机制是**文档指纹**（内容 hash，SHA-256）。每次索引前扫描源目录，与指纹表 diff 出**五态**（边界处理详见 §2.5）：

| 状态 | 判定 | 处理 |
|------|------|------|
| **新增** | 指纹表中无此文件 | 走完整索引流程 |
| **更新** | 文件在表中，但指纹变了 | 先建新后删旧（§8.3） |
| **删除** | 指纹表中有，但源文件没了 | 清理其所有 chunk |
| **移动** | 指纹已在表、但路径变 | 只更 source_path，不重索引 |
| **未变** | 指纹相同 | **跳过**（省时省钱） |

**关键细节**：指纹必须基于**内容**（hash），不能只看 mtime——mtime 会因「另存为」等操作变化但内容没变，误判为更新。

### 8.3 增量 upsert 工程实现

- **新增**：解析 → 分块 → embedding → 写向量库 + BM25 + 更新指纹表
- **更新**：**先建新后删旧**——新版走完整流水线得新 chunk_ids，确认成功后再删旧 chunk_ids（避免空窗丢数据，详见 §2.6 / §2.10）
- **upsert 语义**：存在则替换、不存在则插入——靠 doc_id / chunk_id 定位

### 8.4 删除同步（孤儿清理）⭐

源文件删除后，要清理它在索引里的所有痕迹：

- 按 doc_id 删除：向量库中该 doc 的所有 chunk、BM25 中对应项、embedding 缓存
- **孤儿检测**：定期扫描索引，清理「源文件已不存在但 chunk 还在」的孤儿数据（异常中断会留下孤儿）

孤儿数据是隐性 bug——占用空间、可能被错误召回，必须定期清理。

### 8.5 索引持久化的各层

构建管线产生的多个产物，都要持久化：

| 产物 | 持久化方式 |
|------|-----------|
| 向量库 | Chroma `PersistentClient` 落盘 |
| BM25 索引 | pickle 到磁盘（或 bm25s 原生持久化） |
| embedding 缓存 | sqlite / 磁盘 KV |
| 元数据 | 随向量库 metadata 字段 |
| 指纹清单 | JSON / sqlite 文件 |
| 父块 KV | 磁盘文件（父子分块的父块存储） |

启动时从磁盘加载，索引跨会话复用。

### 8.6 一致性与部分失败处理 ⭐ —— 工程难点

增量是「先建新后删旧」的多步操作，**中间失败会导致不一致**（如删了旧 chunk 但插入新 chunk 时崩溃 → 数据丢失）。深入策略见 **§2.9（异常恢复：幂等重放 + 写入前清残留）** 与 **§2.10（中间态一致性：status 标记 + 查询过滤）**，要点：

- **先插新、后删旧**：避免空窗丢数据；用新 version 区分新旧
- **幂等 + 重试 + 操作日志**：崩溃可恢复 / 续跑
- **多索引协同**：向量库 + BM25 一起成功或回滚（§6.7）

### 8.7 版本管理与回溯（可选）

- `version` 字段：文档多版本共存（旧版不立即删，支持「按版本查询」）
- 时间戳：支持「某时间点知识库」的快照查询
- **代价**：存储翻倍，第一版可不做；当有审计 / 合规需求时再引入

### 8.8 监控与运维

- **索引健康**：chunk 总数、向量数、孤儿率、指纹表与源目录的差异
- **重建策略**：定期**全量校准**（如每周），纠正累积的微小不一致（增量难免有漏网）
- **性能监控**：索引大小、查询延迟、embedding 调用量 / 成本

### 8.9 agents_rag 落地

```
启动: 加载指纹清单 + 各索引(向量库/BM25/缓存/父块KV)
扫描 data/raw/ → 指纹 diff → 四态分流:
  新增: 完整索引流程
  更新: 先插新(新version) → 后删旧 → 更新指纹
  删除: 按 doc_id 清理所有索引 + 指纹表
  未变: 跳过
定期: 全量校准 + 孤儿检测
```

### 8.10 持久化与增量层常见陷阱

| 陷阱 | 后果 |
|------|------|
| 用 mtime 而非内容指纹判变更 | 误判更新、白白重算 |
| 先删后插，中间失败 | 数据丢失 |
| 删除不同步多索引 | 向量与 BM25 不一致 |
| 不清理孤儿 | 空间泄漏、错误召回 |
| 无全量校准 | 增量误差累积 |
| 无 embedding 缓存配合增量 | 未变 chunk 也重算 |

### 8.11 多租户隔离

一个 RAG 服务多租户（部门 / 客户），数据要隔离——A 不能查到 B 的文档。

| 方式 | 做法 | 隔离 | 管理 |
|------|------|------|------|
| **物理隔离**（per-collection） | 每租户一个 collection | 强 | collection 多、小租户浪费 |
| **逻辑隔离**（tenant_id 过滤） | 共用 collection，chunk 带 `tenant_id`，检索 where 过滤 | 靠过滤 | 简单省资源，**要确保不漏过滤** |
| **混合** | 大租户独立 collection，小租户共用 + 过滤 | — | 平衡 |

> agents_rag 第一版可能单租户；多租户倾向 `tenant_id` 过滤 + 大租户独立 collection。**逻辑隔离最大风险是漏过滤导致泄露**，必须在检索层强制。

### 8.12 零停机索引迁移（蓝绿）

换向量库（Chroma → Qdrant）或 embedding 模型升级（§6.13）不能停服时：

- **绿**（旧）继续服务查询；**蓝**（新）后台全量构建
- 构建完**原子切换**（`current_index` 指针）→ 下线绿（保留可回滚）
- **双写期**（可选）：切换前新增文档同时写新旧

**与 §2.10 机制统一** ⭐：零停机蓝绿 = §2.10 update 的「新旧并存 → 原子切换 → 下线旧」，同一套机制从 chunk 级放大到索引级。

### 8.13 灾备与恢复

索引损坏 / 机器故障 / 误删的应对：

**备份**：向量库快照（定期 dump）/ 源数据 + 注册表（`data/raw` + 注册表，重建种子）/ embedding 缓存（加速重建）。

**恢复**：快照恢复（快，分钟级）；或 `data/raw + 注册表` 全量重建（慢但可靠，缓存加速）。

**RPO / RTO**：RPO（可丢数据量）取决于备份频率；RTO（恢复时间）快照分钟级、全量重建小时级。

**落地**：定期快照 + `data/raw` + 注册表备份（异地）；优先快照恢复，不行再全量重建。

**核心 takeaway**：增量维护靠**内容指纹 diff（四态分流）+ 先插后删的一致性策略 + 多索引同步 + 孤儿清理 + 定期全量校准**。这是知识库能长期低成本演进的根基。运维进阶上，**多租户靠隔离防泄露、零停机靠蓝绿、灾备靠快照 + 双备份**。

## 9. 知识库质量保障（最常被忽视）⭐

知识库构建不是「跑一次就完」，要有**评估闭环**。核心是「召回测试集 + recall@k + 调参闭环」，辅以分块抽检、embedding 健康度，并与查询侧 RAGAS 互补。

### 9.1 评估手段全景

| 手段 | 评什么 | 阶段 |
|------|--------|------|
| **召回测试集 + recall@k** | 检索找得准不准 | 构建（核心） |
| 分块抽检 | chunk 语义完整否 | 构建 |
| embedding 健康度 | 向量区分度 | 构建 |
| 参数迭代 | 调 chunk_size / overlap / top_k | 构建 |
| RAGAS | 端到端答得好不好 | 查询（互补） |

### 9.2 召回测试集（核心）⭐

```python
EvalItem:
    query: str                       # 查询
    relevant_chunk_ids: set[str]     # ★ 应召回的 chunk（或 doc）id
    query_type: 事实/对比/探索/多跳   # 分类型看召回
```

**查询来源**（呼应 §12.1.5 评测真实性）：真实日志采样（最佳）/ 从文档生成 Q-A 对（LLM 半自动，起步快）/ 专家编写（偏差大，仅冷启动）。

**相关 chunk 标注**（最耗时）：对每个 query，在已索引 chunk 池里人工定位应召回的 chunk；多人标注 + 一致性检验。

**规模**：冷启动 30–50 条（够调参）；成熟几百条（统计显著）。

### 9.3 recall@k 计算

```
recall@k = |检索top_k ∩ 相关chunk| / |相关chunk|   （应召回的，召回了多少）
```

```python
def eval_recall(eval_set, retriever, k):
    recalls = []
    for item in eval_set:
        retrieved = retriever.retrieve(item.query, k=k)
        hit = len(set(retrieved.ids) & item.relevant_chunk_ids)
        recalls.append(hit / len(item.relevant_chunk_ids))
    return mean(recalls)
```

**相关指标**：precision@k（检索中相关比例）、MRR（排名质量）。**recall@k 是调参主指标**（召回是 RAG 瓶颈）。

### 9.4 调参闭环（chunk_size / overlap / top_k）⭐

```
固定评测集 → 变一个参数（如 chunk_size ∈ {256,384,512,768}）
         → 每组建索引、跑 recall@k → 选最高且语义完整
```

**联动**（呼应 §5.6）：chunk_size ↔ top_k（块小多召回、块大少召回）；chunk_size ↔ overlap（10–20% 防边界丢失）。先单参数扫，再联合微调。

**自动化**（避免手动）：参数网格脚本 + 评测 + 报表 → 选最优。

### 9.5 其他评估手段

- **分块抽检**：人工看一批 chunk 是否语义完整、无截断（定性）
- **embedding 健康度**：相关 / 不相关对的相似度分布是否分离（分得开才好用）

### 9.6 工具

| 选项 | 适用 |
|------|------|
| 自建脚本（Python） | 第一版首选，灵活 |
| RAGAS | 端到端（faithfulness / relevancy + context recall） |
| 框架工具 | LlamaIndex / FlagEmbedding 评测模块 |

### 9.7 agents_rag 落地

- **第一版**：30–50 条评测集（专家编 + 日志采样）+ 自建 recall@k 脚本 + chunk_size / overlap / top_k 调参闭环
- **进阶**：RAGAS 端到端 + 真实日志持续扩充评测集（§12.1.5）

### 9.8 质量保障常见陷阱

| 陷阱 | 后果 |
|------|------|
| 无评测集，靠感觉调参 | 玄学、不可复现 |
| 评测集太小 | 统计不显著 |
| 评测集偏离真实分布 | 过拟合、线上差（§12.1.5） |
| 只看端到端、不分环节 | 定位不了问题 |
| 相关 chunk 标注不准 | recall 算错 |

**核心 takeaway**：质量保障核心是**召回测试集 + recall@k + 调参闭环**（自动化参数网格），辅以分块抽检 / embedding 健康度，端到端靠 RAGAS 互补。recall@k 是调参主指标，参数联动一起扫。

## 10. 常见陷阱清单（避坑）

| 陷阱 | 后果 |
|------|------|
| 一刀切固定大小分块 | 切坏句子 / 表格，召回残缺 |
| 忽略表格 / 结构 | 表格变乱码，领域数据丢失 |
| 不带元数据 | 无法引用溯源 |
| 只建向量索引 | 专有名词 / 编号召回差 |
| 不做增量 | 数据一变就全量重建，慢且贵 |
| embedding 无缓存 | 重复计算，成本高、不可复现 |
| 解析质量差 | 上限被锁死，下游再优化也无效 |
| 过度清洗 | 删掉有用信息（编号、符号、缩进） |
| 先删后建（update 空窗） | 中间态丢数据 / 脏读（§2.10） |

## 11. 小结：知识库构建配方（agents_rag）

```
data/raw/ (指纹 diff)
   → ParserRouter (Docling → MinerU → OCR 降级)
   → 清洗归一化
   → 结构感知分块 + 父子分块 (带 page/heading/parent_id/section_path)
   → embedding-3 批量向量化 (缓存 + 限流)
   → Chroma 向量库 + BM25 索引 (双索引)
   → 指纹清单持久化 (支持增量)
   → 召回测试集 + 抽检 (质量闭环)
```

核心取舍：**在「解析」和「分块（父子）」上投入最多**（质量杠杆）；向量和检索用成熟方案标准化。

> 单环节的工程 / 算法 / 评测待深入点已**全部回填到 §2–§9 正文**（父子分块代码级 §5.14、ParserRouter §3.9、表格保真 §3.10、跨页表 §3.11、稀疏向量 §6.12、索引漂移 §6.13、多租户 / 零停机 / 灾备 §8.11–8.13、实体规范化 / PII §4.7–4.8、元数据抽取 / 权限 §7.9–7.10、评测体系实战 §9）。本节集中收录**多环节领域核心难点**（已展开详述）。

### 12.1 多环节领域核心难点（开放问题，贯穿多环节）⭐⭐⭐

| 核心难点 | 贯穿环节 | 难点本质 |
|---------|---------|---------|
| **多模态元素**（图表 / 公式 / 流程图进入检索） | 解析 + 分块 + 检索 | 纯文本 RAG 丢失关键信息；多模态 embedding / 描述生成不成熟 |
| **上下文连贯性**（指代断裂 / 跨块推理） | 分块 + 检索 + 生成 | 分块必然切断语境；父子分块只部分缓解 |
| **垂直领域适配**（embedding / LLM 对术语不友好） | 向量化 + 生成 | 通用模型对领域黑话不友好；微调成本高、数据难获取 |
| **解析置信度 + HITL**（人工介入） | 解析 + 运维 | 解析不可能 100% 准；如何量化置信、何处引入人工校验 |
| **评测真实性**（评测集是否反映真实分布） | 全局 | 过拟合评测 ≠ 线上好 |
| **成本 - 质量平衡** | 全局 | embedding / rerank / 解析 / LLM 都是成本点；有限预算最大化质量 |

#### 12.1.1 多模态元素进入检索（解析 + 分块 + 检索）

企业文档大量含图表 / 公式 / 流程图 / 示意图，承载关键信息但非文本。纯文本 RAG 丢失核心信息，多模态检索又不成熟。

**处理方案谱（务实 → 原生）**：

| 方案 | 做法 | 优 | 缺 |
|------|------|----|----|
| **A 转文本描述**（主流） | 多模态 LLM 生成图表文字描述 → 入库 | 复用文本管线、改动小 | 丢精确细节、质量依赖模型 |
| **B 多模态 embedding** | 图文映射同一向量空间（CLIP 等） | 原生多模态 | 模型质量、中文、整合复杂 |
| **C 描述 + 原图双索引** | 描述负责召回、原图回传给多模态 LLM | 互补、精确 | 工程重 |
| **D 表格 / 公式专用** | 表格转 Markdown（§3.10）、公式转 LaTeX（§3.2） | 已覆盖 | — |

**进阶**：方案 A 可生成**多角度描述**（概述 + 详细数据 + 关键结论）提升召回。

**关键挑战**：描述质量（数值 / 趋势准确性）、成本（每图调多模态 LLM）、图文关联（图 + 说明文字一起检索）、描述与原图一致性。

> agents_rag 第一版**非目标**，但解析层预留图表识别 + 提取说明文字；后续走方案 A（描述生成），成熟后升级 C。

**核心 takeaway**：多模态是最难的开放问题；务实路径是**方案 A（多模态 LLM 转文本描述）**，要精确升级到**方案 C（描述 + 原图双索引）**。

##### 实施环节归属

多模态处理**全部在构建管线（离线索引）完成**，查询时只检索 + 回传。**解析环节（§3）是所有方案的共同入口**——必须先识别图表、提取图像 / 区域。各方案改动落点：

| 方案 | 解析 §3 | 分块 §5 | 向量化 §6 | 元数据 §7 | 持久化 §8 |
|------|---------|---------|-----------|-----------|-----------|
| **A 描述** | 识别图表 + **生成描述** | 描述分块 | 描述 embedding | 标 block_type | — |
| **B 多模态 emb** | 识别图表 + **提取图像** | 图作原子块 | **图像多模态 embedding** | 图引用 | 原图存储 |
| **C 双索引** | 识别图表 + 提图 + **生成描述** | 描述分块 | 描述 embedding + 图存储 | 图引用关联 | 原图存储 |

- **方案 A 的「轻」**：改动只在解析（加描述生成），后续零改动复用文本管线
- **方案 B/C 的「重」**：跨越解析 + 向量化 + 元数据 + 持久化
- 落地顺序：先打通解析层图表识别 → 方案 A → C → B

##### 图像原始信息保存（方案 B/C）

图像是二进制 blob，向量库不是文件存储，需**单独存储 + 引用关联**——本质是和「文档注册表 + chunk 管理」**平行的一套资产管理**。

**存储介质**：本地文件系统起步 → 对象存储规模化（DB BLOB 不推荐）。

**图像注册表**（类比 §2.3 DocumentRecord）：

```python
ImageRecord:
    image_id: str           # 内容 hash 或 uuid，主键
    doc_id: str             # 来源文档
    source_path: str        # 存储路径 / URL
    page: int; bbox: tuple
    caption: str            # 说明文字
    description: str        # 方案 A/C 生成的描述
    content_hash: str       # 图像内容 hash
    created_at: datetime
```

**关联**：`chunk.metadata.image_ref = image_id`（文本世界 ↔ 图像世界的桥梁）。

**回传**：命中图表 chunk → `image_ref` → 取原图 → 回传多模态 LLM 看图。

##### 存储路径：文档维度（推荐）

图像存储路径用**文档维度**而非平铺：

```
storage/images/<doc_id>/<image_id>.<ext>      # ✅ 文档维度（推荐）
storage/images/<image_id>.<ext>               # 平铺（仅图高频跨文档复用时才值）
```

| 维度 | 平铺 `<image_id>` | 文档维度 `<doc_id>/<image_id>` |
|------|-------------------|------------------------------|
| 跨文档去重 | ✅ | ❌（同图多文档存多份） |
| 文档删除 | 复杂（refcount 判归零） | ✅ 删整个 `doc_id/` 目录 |
| 文档更新 | 复杂（逐图判引用） | ✅ 目录级重建 |
| 孤儿清理 | 复杂（扫全库） | ✅ 目录没了即删 |
| refcount | 需要 | 不需要 |

**澄清**：文档维度**不是失去增量**，而是失去**跨文档去重**。`content_hash` 从「跨文档去重」转为「**文档内增量**」——文档更新时比对该文档新旧图的 `content_hash`，未变的复用（描述 / embedding 不重算）。

> 选择依据：企业文档里图跨文档复用通常很少，去重收益小；文档是 RAG 管理单元，目录级操作简单收益大。**故选文档维度**。仅当图高频跨文档复用（如标准件图库）才值得平铺 + refcount。

#### 12.1.2 上下文连贯性（指代断裂 / 跨块推理）

分块**必然切断语境**：chunk 内的「它 / 该产品」可能指代别处的实体，检索只召回当前 chunk，LLM 看不到指代对象 → 理解错或幻觉。

**问题表现**：① 指代断裂（指代对象在别的 chunk）② 跨块推理（答案要结合多 chunk）③ 上下文缺失（chunk 缺宏观语境）。

**缓解方案谱**：

| 方案 | 做法 | 代价 |
|------|------|------|
| **A 父子分块**（§5.5/5.14） | 子块检索、父块回传 | 指代对象可能仍不在父块 |
| **B 上下文增强** | 回传附 section_path + 相邻 chunk | 仅缓解，非根治 |
| **C 指代消解** | 把「它」替换成指代对象 | 成本高、可能改错 |
| **D Contextual Retrieval** ⭐ | LLM 给 chunk 生成上下文前缀再 embedding | 索引时一次性成本 |
| **E 子问题分解**（查询侧） | 复杂问题拆子问题分别检索 | 放查询管线 |

##### Contextual Retrieval 实施（方案 D，高性价比）

给每 chunk 用便宜 LLM 生成「上下文前缀」，拼接后 embedding（+ BM25）。Anthropic 实测累计降检索失败率约 67%（与 BM25 + rerank 正交叠加）。

**步骤**：① 准备文档 / section 背景 → ② 逐 chunk 调便宜 LLM（Flash）生成 context → ③ `indexed_text = context + chunk` → embedding + BM25 → ④ 检索 + 回传。

**成本控制**：便宜模型 + 缓存（chunk 不变则 context 不变，增量复用）。

##### 大文件：section 级 context（非全局 summary）

大文件下全局 summary 泛化、共享、无区分度。**context 的区分性来自 chunk 所在 section + 位置 + 主题**，不是全局 summary。正确做法是 section 级 context：

```
prompt 输入: doc_title + section_path + section_content（局部聚焦）+ chunk
LLM 生成: 该 chunk 在 section 语境下的 context
```

复用 §3.4 章节树定位 section，零额外解析成本。

##### 检索用、回传不用（职责分离）⭐

context 是 LLM 生成的概括，混入回传会失真。**检索内容和回传内容分离**：

| 阶段 | 内容 | 目的 |
|------|------|------|
| 检索（embedding / BM25） | context + chunk | 召回准 |
| 回传（生成 LLM） | chunk 原文 + 父块 + section_path | 保真 |

- context 只负责**检索定位**，回传时退场
- 生成的上下文连贯靠**原文手段**（父块、section_path），不靠 context
- context 要约束为**客观定位**（「位于第 3 章销售分析」），禁止**内容概括 / 评价**（「表现优异」）——从源头降失真

> 普适原则：**检索优化召回（可加 context / 扩展），回传优化保真（用原文）**，两者内容可分离。

##### agents_rag 定位

- **第一版**：父子分块（A）+ section_path（B）—— 基础缓解，已覆盖
- **进阶（强推）**：Contextual Retrieval（D）—— section 级 context、检索用回传不用、Flash 模型 + 缓存
- 指代消解（C）：成本高，暂不做；跨块推理（E）放查询侧

**核心 takeaway**：上下文连贯是分块固有代价，无法根治只能缓解。靠**分层**：父子分块 + section_path（基础）→ Contextual Retrieval（高性价比进阶，砍一半检索失败）→ 查询侧子问题分解（复杂推理）。关键是**检索 / 回传职责分离**——检索加 context 提召回、回传用原文保真。

#### 12.1.3 垂直领域适配（向量化 + 生成）

通用 embedding / LLM 在专业领域（医疗 / 法律 / 金融 / 工业）打折：术语黑话（通用模型没见过）、同词异义（「容器」IT vs 化工）。适配两层：**embedding 适配（检索瓶颈）+ LLM 适配（生成理解）**。

##### 方案谱

| 方案 | 做法 | 成本 | 效果 |
|------|------|------|------|
| **A prompt 工程** | 系统 prompt 注入领域背景 / 术语表 | 轻 | LLM 改善，embedding 没动 |
| **B 术语表 + 检索扩展** ⭐ | 领域同义词表，检索时扩展 | 中 | 检索改善（性价比首选） |
| **C 领域 embedding** | 选领域预训练 / 微调 | 重 | 检索大幅提升 |
| **D LLM 适配** | RAG（本职）/ 微调（重） | 轻 / 重 | LLM 领域理解 |
| **E 混合** | 通用 embedding + 术语表；LLM 靠 RAG | 中 | 推荐 |

##### 各方案实施（核心逻辑 + 关键模型）

**A prompt 工程**：系统 prompt 注入领域背景 + 术语表（按查询动态注入）。无模型，配合 GLM-4.5。只影响生成不动检索。

**B 术语表 + 检索扩展** ⭐：术语表 `{标准词: [别名/缩写]}`，来源（专家整理 / 语料挖掘 / 领域词库如 UMLS）。query 分词 → 查表加同义词 → 多 query 检索合并。关键：**jieba 自定义词典**（分词层）+ NER 辅助。

**C 领域 embedding**（两路）：
- 路 1 选预训练：**BGE-M3**（中文友好、稠密 + 稀疏 + 多向量）、bge-large-zh、领域专用（医疗 BioBERT 系）。零微调。
- 路 2 微调：**sentence-transformers** + 对比学习（InfoNCE），领域正负样本对，GPU 微调。数据门槛高。

**D LLM 适配**：
- RAG（推荐）：检索领域文档喂 GLM-4.5，**就是 RAG 本身**，不微调。
- 微调（最后）：开源 LLM（Qwen / LLaMA）+ **LoRA**（轻量）/ **LLaMA-Factory**、**PEFT**，领域 Q-A 数据 SFT。闭源 GLM 微调受限，优先 RAG。

##### 关键判断

1. **RAG 本身就是领域适配的轻量方案**（不用微调 LLM）
2. **embedding 适配 > LLM 适配**（检索是瓶颈）
3. **术语表扩展是性价比首选**
4. **微调是最后手段**（数据 + 成本门槛）

##### 关键模型 / 工具

| 用途 | 模型 / 工具 |
|------|------------|
| 通用 embedding | 智谱 embedding-3（底层走 OpenAI 兼容协议，可换平台） |
| 中文 / 领域 embedding | **BGE-M3**、bge-large-zh |
| embedding 微调 | **sentence-transformers** + InfoNCE |
| 通用 LLM | GLM-4.5（+ RAG） |
| 可微调 LLM | Qwen、LLaMA |
| LLM 微调 | **LLaMA-Factory**、**PEFT (LoRA)** |
| 分词 + 术语 | **jieba** 自定义词典 |

##### agents_rag 落地

| 层 | 第一版 | 进阶 |
|----|--------|------|
| embedding | embedding-3 + **术语表扩展（B）** | 区分不足 → BGE-M3 / 领域 embedding（C） |
| LLM | GLM-4.5 + prompt 注入术语（A）+ RAG | RAG 不够 → 开源 LoRA（D2） |
| 微调 | 不做 | 最后手段 |

**核心 takeaway**：领域适配优先 **embedding**（检索瓶颈），靠**术语表 + jieba 词典 + 检索扩展**（性价比首选）；LLM 靠 **RAG 本身**（不微调）。微调（C/D）是检索和术语扩展都不够时的升级。RAG 价值正是「不用微调就让通用模型回答领域问题」。

#### 12.1.4 解析置信度 + HITL（解析 + 运维）

解析不可能 100% 准。错误解析进索引 → 检索到错的、生成基于错的（脏数据污染）。需识别「低置信度」解析，引入人工校验（HITL）形成质量闭环。

##### 解析置信度（多信号聚合）

- **OCR 置信度**：引擎返回（RapidOCR / PaddleOCR 有 `confidence`）
- **表格识别置信**：模型对结构
- **版面分析置信**：区域分类
- **启发式信号**（§3.7）：文本量异常、乱码占比、结构合理性

chunk / 文档级置信度 = 各信号**加权聚合**，低于阈值 → 标记复核。

##### HITL 触发与流程

**触发**：低置信（< 阈值）/ **关键内容强制**（法律条款、医疗剂量、金额）/ 抽样复核。

```
解析 → 置信度评估 → 分流:
  高置信 → 自动入库
  低置信/关键 → 复核队列 → 人工标注/修正 → 回写索引（重新 embedding）
```

**工具**：标注界面（原文 vs 解析对照，高亮差异）+ 批量审核。

**反馈闭环**：人工修正反哺解析调优（收集常错点，针对性调 parser）。

##### 成本权衡

HITL 增人力，但避免脏数据（错数据代价远高于人力）。**只对低置信 + 关键内容，不全量人工**。

##### agents_rag 定位

- **第一版**：解析质量信号（§3.7）+ 低置信标记（chunk 元数据带 `parse_confidence`）
- **进阶**：HITL 复核队列（低置信 + 关键）+ 标注界面
- **反馈闭环**：修正反哺解析

**核心 takeaway**：解析置信度靠**多信号聚合**，HITL 对**低置信 + 关键内容**人工复核并回写，形成「解析 → 评估 → 分流 → 人工 → 回写 → 反哺」质量闭环。不全量人工（成本），靠置信度精准触发。这是把 §3.7「质量评估」升级为「可干预闭环」。

#### 12.1.5 评测真实性（全局）

评测集是调优仪表盘，但仪表盘可能失真：不反映真实查询分布、过拟合评测 ≠ 线上好、规模小不显著、陈旧过时。

##### 评测真实性的四个维度

| 维度 | 要求 |
|------|------|
| **查询分布真实性** | 反映真实用户查询，**真实日志 > 专家编造** |
| **标注质量** | 专家标注 + 多人一致性（Cohen's kappa） |
| **多样性与规模** | 覆盖主题 / 类型 / 难度，规模够 |
| **时效性** | 定期更新 |

##### 评测方法（多层）

| 方法 | 特点 |
|------|------|
| **自动（RAGAS）** | faithfulness / relevancy / precision / recall，快，依赖标准答案 |
| **人工** | 最准但慢 |
| **LLM-as-judge** | 折中（注意 judge 偏差） |
| **在线** | 线上真实查询 + 用户反馈，最真实但稀疏 / 滞后 |

##### 评测真实性保障

**评测集构建**：真实日志采样（去敏）→ 专家标注标准答案 + 相关 chunk → 多人一致性检验 → 覆盖类型 / 主题 / 难度。

**防过拟合** ⭐：不只看评测分数，结合线上反馈；评测集定期轮换 / 扩充；**hold-out 测试集**（不参与调优，仅最终验证）。

**多层评测**：分环节（recall@k / precision / faithfulness）+ 端到端（RAGAS）+ 线上（真实分布 + 反馈）。

##### agents_rag 定位

- **第一版**：小评测集（专家编 + 真实日志采样）+ RAGAS 自动回归
- **进阶**：真实日志持续扩充 + 人工评测 + 线上反馈闭环
- **防过拟合**：hold-out + 定期更新

**核心 takeaway**：评测真实性靠「**真实日志采样 + 专家标注 + 多人一致性 + 多层评测 + 防过拟合**」。**真实性 > 分数**——反映真实分布的小评测集，比偏离分布的高分评测集有价值得多。

#### 12.1.6 成本-质量平衡（全局）

RAG 全链路都是成本点，企业质量要求高。有限预算下最大化质量是工程艺术。

##### 成本点全景（索引侧 vs 查询侧）

| 环节 | 成本 | 频率 | 优化点 |
|------|------|------|--------|
| 解析（OCR / 多模态描述） | API / 算力 | 索引时 | 按需启用 |
| embedding | token | 索引时 | **缓存** |
| Contextual Retrieval | LLM 调用 | 索引时 | 便宜模型 + 缓存 |
| 存储 | 向量库 / 图像 | 持续 | 维度 / 生命周期 |
| rerank | API | **查询时** | 按需 |
| LLM 生成 | token | **查询时** | 模型分层 |
| HITL | 人力 | 复核时 | 低置信 + 关键 |

> **查询侧成本随查询量线性增长**，比索引侧（一次性 + 增量）更需优化。

##### 平衡策略

1. **缓存（最有效）**：embedding 缓存（§6.5）、Contextual Retrieval 缓存、**查询结果缓存**（语义缓存）
2. **便宜模型分层**：辅助任务（context 生成、分类）用 **Flash / Haiku**，主生成用 GLM-4.5，rerank 用 API
3. **按需启用进阶**：Contextual Retrieval / rerank / HITL / 多模态 按需，非默认全开
4. **批量 + 异步**：批量 embedding 摊薄、非实时异步
5. **存储 / 检索优化**：维度（2048→1024 省）、HNSW、冷数据归档

##### 成本-质量曲线（80/20）⭐

- **基础方案**（embedding-3 + BM25 + 父子）低成本覆盖 **~80% 质量**
- **进阶**（Contextual Retrieval / rerank / 领域适配 / HITL）提升剩余 20%，但**成本陡增、边际递减**
- 找**性价比拐点**：按业务价值决定投入多深

##### agents_rag 定位

- **第一版（基础区）**：embedding-3 + BM25 + 父子分块——成本可控，80% 质量
- **缓存**：embedding + 查询结果
- **模型分层**：Flash 用于辅助
- **按需进阶**：rerank 默认开；Contextual Retrieval / HITL / 多模态按需

**核心 takeaway**：成本-质量平衡靠「**缓存（最有效）+ 模型分层 + 按需启用进阶 + 批量异步 + 存储优化**」。遵循 80/20——基础方案低成本覆盖 80% 质量，进阶按业务价值逐步投入。
