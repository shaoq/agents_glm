## 0. 核心认知

查询管线的本质，是把**「用户问题」变成「带引用的可信回答」**。在 RAG 体系里这条线叫**查询管线（query pipeline）**，是在线 / 实时过程。

三个关键判断：

1. **检索是瓶颈**：构建再好，查询侧召回 / 重排不到位，照样答不好。
2. **70% 法则（查询侧）**：质量风险集中在「**检索（召回 + rerank）**」和「**生成（prompt + grounding）**」。
3. **五段总纲**：
   - **查询改写**决定「拿什么去检索」
   - **检索 + 重排**决定「找得准不准、排得对不对」
   - **上下文构建**决定「喂给 LLM 的精不精」
   - **生成**决定「答得稳不稳（抗幻觉）」
   - **引用溯源**决定「信不信得过」

一个常被忽视的认知：**原始用户问题，往往不是最适合拿去检索的 query**——这是查询管线最常见的浪费。

---

## 1. 全景：查询管线的 7 个环节

```
用户问题 → ①查询理解/改写 → ②检索(向量+BM25 混合) → ③重排序(Rerank)
        → ④上下文构建 → ⑤生成 → ⑥引用溯源/抗幻觉 → ⑦评估(RAGAS)
```

**两条主线**：①–④ 是「把相关信息找出来并组织好」，⑤–⑥ 是「基于信息生成可信回答」，⑦ 贯穿诊断。

**核心节奏**：召回要**宽**（top_k 大，保证 recall）→ 精排要**严**（rerank 选最相关）→ 上下文要**精**（去重 + 预算 + 父子回传）→ 生成要**守**（grounding + 引用）→ 全程要**测**（RAGAS）。

---

## 2. 查询理解与改写（Query Transformation）

原始 query 往往简短、模糊、口语化、多义，直接召回效果差。

- **查询改写**：LLM 把模糊表述改成更利于检索的关键词式
- **多查询（Multi-Query）**：生成 N 个变体，分别检索后合并（扩召回）
- **子问题分解**：复杂问题拆子问题分别检索（多跳）
- **HyDE**：先让 LLM 生成「假设性答案」，用答案的 embedding 检索（query 与文档表述差异大时极强）
- **Step-Back**：把具体问题抽象到更高层概念再检索

**权衡**：所有改写都增加 LLM 调用（延迟 + 成本），通常简单 query 走快路径、复杂 / 短 query 才启用。

---

## 3. 检索（Retrieval）

### 两路召回
- **向量检索**：语义召回（embedding 相似度）。强在「意思相近」，弱在精确词、型号、编号
- **BM25**：词频 + IDF。强在精确匹配、专有名词，弱在语义

### 混合检索 + 融合
- **RRF（Reciprocal Rank Fusion）**：只用排名、无需调参，鲁棒（推荐默认）
- **加权得分**：需归一化 + 调权，上限略高但工程复杂

### 其他
- **元数据过滤**：pre-filter（先过滤再检索，省且准）/ post-filter（检索后过滤，通用）
- **召回要宽**：top_k 取大（如 20–50），交给 rerank 精选

> 检索用的是构建管线产出的索引（向量库 + BM25），二者在此衔接。

---

## 4. 重排序（Rerank）

向量召回是**粗排**（bi-encoder，快但精度参差）；rerank 是**精排**（cross-encoder，准但慢）。

- **bi-encoder vs cross-encoder**：前者 query/doc 独立编码（可离线建库），后者拼一起联合编码（每对都要前向，准但慢）
- **两阶段必然性**：召回用快的 bi-encoder 扫全库，rerank 用准的 cross-encoder 只处理几十个候选
- **流程**：检索 top_k（如 50）→ Rerank → top_n（如 5–10）
- **选型**：智谱 Rerank API（零运维）/ 本地 bge-reranker（中文友好、需算力）

---

## 5. 上下文构建（Context Building）

检索到的片段不能直接拼进 prompt。

- **去重**：混合检索必重复，按 hash / 相似去重
- **排序**：按 rerank 分数降序；注意 **lost in the middle**（头尾利用好、中间易忽略）
- **token 预算控制**：不超过 LLM 上下文（留生成空间）
- **相关性阈值**：低于阈值的丢弃，宁缺毋滥
- **父子回传**：小块检索命中 → 回传父块（完整上下文，呼应构建管线父子分块）
- **引用编号注入**：`[1] 文本... [2]...`，为溯源打基础

---

## 6. 生成（Generation）

- **Prompt 工程**：
  - **Grounding**：仅基于提供的上下文回答；上下文没有就明确说「未找到」
  - **强制引用**：每个论断后标注 `[编号]`
  - **格式约定**：先结论后展开 / 结构化输出
  - **温度**：事实问答用低温度（0–0.3）减少发散
- **上下文注入**：把构建好的上下文（带编号）注入 prompt
- **多轮对话**：历史压缩 / 只留近 N 轮；每轮用「当前问题 + 检索上下文」生成

---

## 7. 引用溯源与抗幻觉

企业场景硬需求——回答必须能溯源、不能编造。

### 引用溯源
- 每条引用绑定：文档名 + 页码 + 原文片段（靠构建管线的元数据 source/page/char_span）
- UI 层可点击跳转原文（后期）

### 抗幻觉（多层防御）
1. **Prompt 层**：强约束「仅基于上下文、不知就说不知」
2. **后处理校验**：生成的 `[编号]` 必须命中检索结果，否则剔除 / 标记（CitationChecker）
3. **兜底**：检索为空或置信度低 → 返回「未找到相关内容」，绝不硬编
4. **可选**：faithfulness 二次校验（LLM 判断答案每句是否都有上下文支撑）

> 这是企业 RAG 与玩具 demo 的分水岭：**敢不敢亮来源、能不能在不知道时认怂**。

---

## 8. 评估（RAGAS）

没有评估的 RAG 是玄学。RAGAS 4 个核心指标分别诊断不同环节：

| 指标 | 诊断 | 对应环节 |
|------|------|---------|
| **Context Recall** | 该召回的召回到了吗 | 检索（+构建） |
| **Context Precision** | 召回的相关吗 / 排序好吗 | 检索 + Rerank |
| **Faithfulness** | 答案忠于上下文吗（有无幻觉） | 生成 + 抗幻觉 |
| **Answer Relevancy** | 答案切题吗 | 生成 + 改写 |

> 用法：指标下降时能**定位到环节**（faithfulness 掉 → 抗幻觉 / prompt；context recall 掉 → 检索或构建分块）。需要评测集（问题 + 标准答案 + 相关文档）。

---

## 9. 常见陷阱清单（避坑）

| 陷阱 | 后果 |
|------|------|
| 原始 query 直接检索 | 模糊 / 口语化 query 召回差 |
| 只用向量、无 BM25 | 专有名词 / 编号召回漏 |
| 检索即终、无 rerank | precision 上不去 |
| 上下文无脑拼接、无去重 / 预算 | token 浪费、lost-in-the-middle |
| 不做引用校验 | 模型乱标编号、可信度崩 |
| 无评估集 | 调参靠玄学、回归无保障 |
| 多轮把历史当事实 | 历史幻觉累积 |

---

## 10. 小结：查询管线配方（agents_rag）

```
用户问题 → (查询改写/HyDE 可选)
        → 向量召回 + BM25 召回 → RRF 融合 → 智谱 Rerank 精排
        → 上下文构建(去重 + token 预算 + 父子回传 + 引用编号)
        → GLM-4.5 生成(grounding + 强制引用)
        → 引用校验 → [带引用来源的回答]
        (RAGAS 贯穿评估)
```

核心取舍：**召回宽、精排严、上下文精、生成守、全程测**。检索与生成是质量主战场。
