## 0. 核心认知

查询管线的本质，是把**「用户问题」变成「带引用的可信回答」**。在 RAG 体系里这条线叫**查询管线（query pipeline）**，是在线 / 实时过程。

三个关键判断：

1. **检索是瓶颈**：构建再好，查询侧召回 / 重排不到位，照样答不好。
2. **70% 法则（查询侧）**：质量风险集中在「**检索（召回 + rerank）**」和「**生成（prompt + grounding）**」。
3. **五段总纲**：
   - **查询改写**决定「拿什么去检索」
   - **检索 + 重排**决定「找得准不准、排得对不对」
   - **上下文构建**决定「喂给 LLM 的精不精」
   - **生成**决定「答得稳不稳（抗幻觉）」
   - **引用溯源**决定「信不信得过」

一个常被忽视的认知：**原始用户问题，往往不是最适合拿去检索的 query**——这是查询管线最常见的浪费。

---

## 1. 全景：查询管线的 7 个环节

```
用户问题 → ①查询理解/改写 → ②检索(向量+BM25 混合) → ③重排序(Rerank)
        → ④上下文构建 → ⑤生成 → ⑥引用溯源/抗幻觉 → ⑦评估(RAGAS)
```

**两条主线**：①–④ 是「把相关信息找出来并组织好」，⑤–⑥ 是「基于信息生成可信回答」，⑦ 贯穿诊断。

**核心节奏**：召回要**宽**（top_k 大，保证 recall）→ 精排要**严**（rerank 选最相关）→ 上下文要**精**（去重 + 预算 + 父子回传）→ 生成要**守**（grounding + 引用）→ 全程要**测**（RAGAS）。

## 2. 查询理解与改写（Query Transformation）⭐

### 2.1 原理：原始 query 往往不是最佳检索 query

核心认知：**原始用户问题，常常不是最适合拿去检索的 query**——这是查询管线最常见的浪费。原因：

- **简短/口语化**：「密码咋办」vs 文档里「账户密码重置流程」
- **多义/指代**：「它的价格」——「它」指什么需消解
- **表述差异**：用户问「怎么重置密码」，文档写「忘记密码处理」（关键词不重叠，向量部分覆盖、BM25 全漏）
- **复杂多跳**：「A 公司供应商的主要客户」需跨文档串联

### 2.2 方案谱（由轻到重）

| 方案 | 做法 | 适用 | 代价 |
|------|------|------|------|
| **查询改写** | LLM 把模糊/口语改成检索友好的关键词式 | 短/模糊 query | 1 次 LLM |
| **多查询（Multi-Query）** | 生成 N 个变体，分别检索后合并（扩召回） | 召回不全 | N 倍检索 + 1 LLM |
| **子问题分解** | 复杂问题拆子问题分别检索 | 多跳推理 | 多次检索 + 1 LLM |
| **HyDE**（强） | LLM 生成「假设性答案」，用**答案的 embedding** 检索 | query 与文档表述差异大 | 1 LLM + 1 embedding |
| **Step-Back** | 把具体问题抽象到更高层概念再检索 | 概念性提问 | 1 LLM |

### 2.3 HyDE 为什么强（单独讲）

向量检索痛点：**query 短而文档长**，query embedding 和文档 embedding 表征不对等，相似度被稀释。HyDE 让 LLM 先「编」一个假设答案（可能不准），但**假设答案和文档都是「答案式长文本」**，表征对等 → embedding 匹配度暴涨。

> 反直觉点：HyDE 的假设答案**事实可以错**（rerank 会纠偏），但**术语/方向不能错**（embedding 跟着偏）——领域场景恰恰术语最易错。

### 2.4 各方案核心难点

| 方案 | 核心难点 |
|------|---------|
| **查询改写** | 改写质量不确定（过度丢原意 / 不足没改善）；中间步骤效果难评估 |
| **多查询** | 变体「多样性 vs 发散」难平衡（雷同=冗余，发散=噪音）；N 路合并去重 |
| **子问题分解** | 分解本身是推理任务（拆错→召回错）；**真多跳需迭代**（子问题依赖前一步答案），非一次分解能解 |
| **HyDE** | **领域偏差致命**（LLM 对垂直术语不熟→假设答案用错术语→embedding 偏）；召回不可解释 |
| **Step-Back** | 抽象层次难把握（太泛/太具体）；适用面窄（只适合概念性提问） |

### 2.5 系统性难点（比方案本身更难、更易踩坑）⭐⭐

这些不是某方案的难点，而是「**把改写插进在线查询管线**」这个决定本身的难点：

| 难点 | 说明 | 严重度 |
|------|------|--------|
| **A 触发判据** | 不是所有 query 都该改写。判断「要不要改写」的信号（短/无实体/指代/口语）难精确；路由错了（简单 query 浪费 / 复杂 query 漏）| ⭐⭐⭐ 最关键 |
| **B 评估闭环** | 改写是中间步骤，效果被下游（检索/rerank/生成）淹没，难归因。RAGAS 诊断端到端，不诊断改写单独。需专项「query→改写后 recall@k」评测集，但很难建 | ⭐⭐⭐ 隐蔽 |
| **C 延迟成本** | 在线实时场景，改写加 1 LLM → 用户感知延迟翻倍。多查询/子问题更重 | ⭐⭐ |
| **D 下游耦合** | 改写喂两路检索，但**向量要语义、BM25 要词**，口味相反。改成关键词式利 BM25 但可能降向量；HyDE 只补向量、BM25 走原 query（两路不对称）| ⭐⭐ |
| **E 多轮指代** | 多轮里「它的价格」需结合历史消解「它」。改写 = 历史压缩 + 指代消解 + 改写三合一；历史幻觉累积 | ⭐ |
| **F 可控/可解释** | LLM 改写是黑盒，方向不可控；改写后 query 与原意差异大时用户困惑；调试难定位（改写 vs 检索谁的错）| ⭐ |
| **G 缓存** | 改写结果可缓存但 query 长尾多样，命中率低；HyDE 假设答案语义缓存复杂 | ⭐ |

### 2.6 agents_rag 落地

| 方案 | 首版 | 理由 |
|------|------|------|
| 查询改写 / HyDE / 多查询 / 子问题分解 | ⏳ 全部后置 | 先打通主线（检索→生成），改写是召回优化；待「短/模糊 query 召回差」的**实测证据** + 评测集就绪后再引入 |

**首版**：query 直接走向量 + BM25 检索（`embed(query)` 与文档同模型 embedding-3，同空间，基础语义召回够用）。改写 / HyDE 是「召回不够时的升级」，非默认。

> 关键顺序：先建评测集（query + 应召回文档），有 recall@k 基线后，再衡量改写收益（规避难点 B）。

### 2.7 查询改写层常见陷阱

| 陷阱 | 后果 |
|------|------|
| 所有 query 无脑改写 | 简单 query 多一次 LLM，延迟 + 成本浪费 |
| 领域文档用 HyDE | LLM 不熟术语 → 假设答案偏 → 召回偏 |
| 多查询不去重合并 | 重复召回、上下文冗余 |
| 子问题一次分解真多跳 | 子问题缺前步答案（「它的客户」无「它」）→ 漏 |
| 改写只产一个 query 喂两路 | 顾向量失 BM25 或反之（下游耦合）|
| 无评测集就上改写 | 收益说不清、调参靠感觉（评估闭环缺失）|

## 3. 检索（Retrieval）⭐⭐

> 检索用的是构建管线产出的索引（向量库 + BM25），二者在此衔接。检索是查询管线的第一硬骨头，也是与知识构建衔接最紧密的环节。

### 3.1 本质：两路召回 + 融合（为什么双路）

知识构建产出两套独立索引（Chroma 向量 + BM25），检索环节**并行查两路 → 融合成统一排序**。两路能力**正交互补**：

| 维度 | 向量召回（embedding-3） | BM25 召回（jieba + rank_bm25） |
|------|----------------------|------------------------------|
| 匹配方式 | 语义相似（意思相近） | 词频 + IDF（精确词匹配） |
| 强项 | 「重置密码」召回「忘记密码」（语义同、词不同） | 「GLM-4.5」「RFC-1234」精确命中 |
| 弱项 | 型号/编号/专有名词（语义向量不够区分） | 同义不同词（「密码」vs「口令」全漏） |

**一句话**：向量管「意思」，BM25 管「精确词」。只用向量 → 专有名词漏；只用 BM25 → 同义漏。**两路互补才是完整召回**。

### 3.2 bi-encoder 粗排（为什么检索后还要 rerank）

向量召回用的是 **bi-encoder**（query/doc 各自独立编码，算余弦相似度）。局限：**独立编码丢失交互信息**，精度有上限。

```
bi-encoder（召回）：query→[vec_q]  doc→[vec_d]  相似度=cos(vec_q,vec_d)   ← 快，doc 可离线建库
cross-encoder（rerank）：query+doc → 拼一起联合编码 → 相关性分数           ← 准，每对一次前向
```

**两阶段必然性**：召回用快的 bi-encoder 扫全库（top_k=20-50），rerank 用准的 cross-encoder 只处理几十个候选（top_n=5-10）。检索负责**宽召回**（保 recall），rerank 负责**严精排**（保 precision）。

### 3.3 RRF 融合（排名融合，鲁棒）

两路各返回一个排序，要合成统一排序。核心选择：**用分数还是排名？**

**加权分数（不推荐）**：`α × normalize(vec_score) + (1-α) × normalize(bm25_score)`——分数量纲不同（余弦∈[0,1]，BM25∈[0,∞)），归一化方式影响结果；权重 α 需调、过拟合。

**RRF（Reciprocal Rank Fusion，推荐）**：

```
RRF_score(d) = Σ 1 / (k + rank_i(d))     # k 通常 60
```

只用**排名**（不用分数），绕开量纲/归一化；无需调权（k=60 通吃）；对异常值不敏感。直觉：**两路都排靠前的文档得分高**（两路共识 = 更可能相关）。

### 3.4 元数据过滤（pre vs post + status）

| 方式 | 流程 | 特点 |
|------|------|------|
| **pre-filter** | 先按 metadata 过滤 → 再向量检索 | 省算力、结果准；需向量库支持（Chroma `where`） |
| **post-filter** | 先检索 top_k → 再按 metadata 过滤 | 通用；但可能过滤后所剩无几（ACL 权限后候选骤减） |

agents_rag：检索时 `where={"status": "active"}` 过滤 update 中间态旧 chunk（pre-filter，复用知识构建写入的 status 字段）。ACL 等强约束也用 pre-filter；时间范围等软筛选可 post。

### 3.5 父子检索 AutoMerging（RRF + Rerank 之后）⭐

**核心时序原则**：检索/融合/精排全在**子块**（检索单元）上做，AutoMerging 在**精排之后**对 top_n 子块做父子回传。

```
① 向量召回（子块 top_k=20）
② BM25 召回（子块 top_k=20）
③ RRF 融合（两路子块排名 → 统一排序）
④ Rerank 精排（子块 cross-encoder → top_n=6）
⑤ AutoMerging（top_n 子块按 parent_id 分组 → ≥threshold 回传父块）  ← 在此
⑥ 上下文构建（父块+子块混合 → 去重 + token 预算 + 引用编号）
```

**为什么必须在 RRF + Rerank 之后**：RRF/Rerank 需要统一的检索单元（子块）。若 RRF 前换父块，两路粒度不一致（父块 vs 子块），排名无法对齐 → 融合失败。AutoMerging 是「回传优化」（子块→父块扩展上下文），不是检索操作。

**AutoMerging 逻辑**：rerank 后的 top_n 子块按 `parent_id` 分组；某父块下 ≥ `merge_threshold`（如 2）个子块命中 → 回传整个父块（`ParentStore.get`，文件点查）；命中少 → 回传子块本身。

### 3.6 top_k 取值（评测驱动 + 联动）⭐

**核心矛盾**：top_k 小 → recall 低（漏）；top_k 大 → precision 降 + Rerank 成本/延迟 + token 浪费。

**判断方法：recall@k 曲线找拐点**——评测集（query + 相关 chunk），不同 k 跑 recall@k：

| k | recall@k | 边际增量 |
|---|---------|---------|
| 10 | 0.78 | — |
| **20** | **0.90** | +0.12 |
| 30 | 0.92 | +0.02 ← 拐点 |
| 50 | 0.93 | +0.01 |

选拐点附近的最小 k（recall 接近上限、成本可控）= 20。

**四个联动关系**（不能孤立调）：

| 联动 | 关系 |
|------|------|
| top_k ↔ chunk_size | chunk 小 → top_k 大（每块信息少、多召回）；**一起扫评测** |
| top_k ↔ top_n | top_k > top_n（召回宽、精排严，给 Rerank 筛选空间） |
| top_k ↔ Rerank 成本 | top_k 大 → 候选多 → API 贵/慢 |
| top_k ↔ token 预算 | 间接（AutoMerging + context_builder 做 token 截断） |

**经验锚点**：向量/BM25 各 20-50（召回），Rerank top_n 5-10（精排后保留）。agents_rag 首版 VECTOR_TOP_K=20 / BM25_TOP_K=20 / RERANK_TOP_N=6。

**场景差异**：事实查询（top_k 偏小，精准命中）；综述/对比/多跳（top_k 偏大，多文档覆盖）。进阶：查询路由动态 top_k。

### 3.7 检索的核心难点

| 难点 | 说明 |
|------|------|
| top_k 选择 | recall vs cost 权衡，评测驱动（见 §3.6） |
| 空召回 | query 与索引完全不相关 → 检索空。必须兜底「未找到」，不硬编 |
| 过滤后候选骤减 | ACL/元数据 post-filter 后 top_k 被过滤光 → 用 pre-filter 或增大 top_k |
| 双路同步 | 向量库与 BM25 的 chunk_id 对齐（构建侧保证）、增删改同步（五态增量保证） |
| BM25 query 分词 | 中文 BM25 必须 jieba 分词，query 与 doc 同分词器 |
| 召回质量评估 | recall@k 主指标，需评测集；无评测 = 靠感觉 |

### 3.8 agents_rag 落地

| 组件 | 首版 | 说明 |
|------|------|------|
| 向量召回 | ✅ | ChromaStore.query（embedding-3 编码 query → 近邻 top_k） |
| BM25 召回 | ✅ | BM25Index.query（jieba 分词 query → BM25 top_k） |
| RRF 融合 | ✅ | k=60，无需调参 |
| status 过滤 | ✅ | `where={"status": "active"}`（pre-filter） |
| 父子 AutoMerging | ✅ | merge_threshold=2，衔接 ParentStore |
| top_k / top_n | 20 / 6 | 经验锚点，评测集就绪后调 |

### 3.9 检索层常见陷阱

| 陷阱 | 后果 |
|------|------|
| 只用向量、无 BM25 | 专有名词/编号召回调 |
| RRF 前做 AutoMerging | 两路粒度不一致（父块 vs 子块）→ 融合失败 |
| top_k 拍脑袋 | recall 或 precision 失衡 |
| post-filter 后候选光 | 权限/过滤后 top_k 所剩无几 |
| BM25 query 不分词 | 中文按字匹配、噪音大 |
| 空召回硬编答案 | 幻觉 |

## 4. 重排序（Rerank）⭐

### 4.1 原理：cross-encoder 为什么比 bi-encoder 准

向量召回是 **bi-encoder**（query/doc 各自独立编码、cos 相似度），**快但丢交互**——query 和 doc 的 token 从不「见面」，word-level 对应关系被压缩丢失。rerank 用 **cross-encoder**：query + doc 拼成一个序列 → Transformer 自注意力让 query 的每个 token「看到」doc 的每个 token → 输出相关性分数。捕捉细粒度交互（如「重置」精确对应「密码恢复」中的「恢复」）。

> 代价：每对 (query, doc) 一次完整前向。全库不可行，但只对召回的几十个候选做，完全 OK。**没有 rerank = 精度上限被 bi-encoder 锁死**。

### 4.2 两阶段架构（必然性）

```
阶段1 召回（bi-encoder）：扫全库 → top_k=20-50（快，保 recall）
阶段2 精排（cross-encoder）：只处理 top_k 候选 → top_n=5-10（准，保 precision）
```

全库 cross-encoder 不现实（N×前向）；召回已缩到几十候选，cross-encoder 几十次前向可接受。**速度与精度的工程平衡**。

### 4.3 选型：智谱 Rerank API vs 本地 bge-reranker

| 维度 | 智谱 Rerank API | 本地 bge-reranker-v2-m3 |
|------|----------------|------------------------|
| 部署 | **零运维**（云端 API） | 需下载模型 + CPU/GPU |
| 延迟 | 网络 RTT（~100-300ms） | GPU 快、CPU 慢 |
| 成本 | 按调用计费 | 免费（但算力/内存） |
| 中文 | 智谱中文原生 | bge 中文友好（BAAI） |
| 隐私 | query + 候选发云端 | **本地不出域** |

**agents_rag 选智谱 Rerank API**：零运维、中文原生、与 embedding/LLM 同生态。代价是候选发云端（涉密文档需本地 bge）。模型 ID 实施时按 [官方文档](https://docs.bigmodel.cn/api-reference/模型-api/文本重排序) 核实。

### 4.4 top_n 选择 + 分数阈值

**top_n**（rerank 后保留）：太小（3）漏相关；太大（15）上下文噪音 + token 浪费。经验 5-10，评测调（同 top_k，靠 precision@n / answer 质量）。首版 top_n=6。

**分数阈值**（可选）：低于阈值的丢弃（宁缺毋滥）。但绝对值因模型/数据而异，首版不做（按 top_n 截断），有评测后加。

### 4.5 精确时序（RRF 后、AutoMerging 前）

```
① 向量召回（子块 top_k=20）
② BM25 召回（子块 top_k=20）
③ RRF 融合 → 子块统一排序
④ Rerank 精排（子块 cross-encoder → top_n=6）   ← 重排序在此
⑤ AutoMerging（top_n 子块 → 按 parent_id 回传父块）
⑥ 上下文构建
```

rerank（④）在 RRF（③）后、AutoMerging（⑤）前——**对子块精排，精排后再父子回传**。rerank 评分的是子块文本（query + 子块），AutoMerging 后换父块是「回传扩展」。不能先换父块再 rerank（粒度不一致，同 §3.5）。

### 4.6 难点

| 难点 | 说明 |
|------|------|
| 成本/延迟 | 每查询一次 rerank API（top_k 候选），在线实时延迟敏感 |
| top_n 选择 | 太小漏、太大噪，评测驱动 |
| 分数阈值 | 绝对值因模型/数据而异，难定通用阈值 |
| 空结果 | rerank 后全极低分 → query 可能不相关 → 兜底「未找到」 |
| 隐私 | API 方案候选发云端；涉密场景需本地 bge |
| 时序 | 必须子块上 rerank → 再 AutoMerging（不能先换父块） |

### 4.7 进阶（首版全后置）⭐

**① 多轮 rerank（邻居扩展）**：首次 rerank top_n1 → 取同 parent/section 的相邻 chunk（未命中的）→ 二次 rerank 确认邻居是否相关 → top_n2。补救召回遗漏。代价：二次 API + 邻居扩展逻辑。与 AutoMerging 互补：AutoMerging = 已知命中扩展上下文（零成本）；多轮 rerank = 发现未命中的邻居（二次 API）。

**② rerank 分数 → context_builder 排序（lost-in-the-middle）**：LLM 对上下文头尾注意力强、中间弱。三种排序：A 降序（简单、首版）、B 头尾高分（优化注意力但破坏引用连贯）、C 文档分组（引用友好但分数不主导）。首版 A（降序）+ 引用编号；上下文变长时再考虑 B。rerank 分数还可做：token 预算分配（高分优先占预算）、引用置信度展示（UI）。

**③ Rerank + Contextual Retrieval 联动**：若知识构建侧上了 Contextual Retrieval（chunk 带 context 前缀，indexed_text = context + chunk），rerank 输入应用 indexed_text（与检索链一致：向量/BM25/rerank 用同一 text）。前提是 context 约束为客观定位（§12.1.2，低失真）。**未上 CR 时无此问题**（原文 = indexed_text，无矛盾）。

### 4.8 agents_rag 落地

| 项 | 首版 | 说明 |
|----|------|------|
| Reranker | 智谱 Rerank API | 零运维、中文原生 |
| 输入 | query + RRF 融合后 top_k 候选 | 子块文本 |
| 输出 | top_n=6 | 按分数截断 |
| 分数阈值 | 不做 | 首版按 top_n 截断 |
| 时序 | RRF 后、AutoMerging 前 | 对子块精排 |
| 排序策略 | 降序 | context_builder 按分数降序 |
| 进阶 | 全后置 | 多轮/lost-in-middle/CR 联动 |

### 4.9 重排序层常见陷阱

| 陷阱 | 后果 |
|------|------|
| 无 rerank（检索即终） | precision 被 bi-encoder 锁死 |
| 对父块做 rerank | 粒度与检索不一致（检索子块、rerank 父块） |
| top_n 拍脑袋 | recall 或 precision 失衡 |
| rerank 后不兜底 | 全低分时硬编答案（幻觉） |
| 涉密文档用 API | query + 候选泄露云端 |
| CR 启用后 rerank 用原文 | 与检索不一致（检索用 indexed_text、rerank 用原文 → 评分偏差） |

## 5. 上下文构建（Context Building）⭐

> 衔接 rerank + AutoMerging 结果 → 生成输入。把检索片段组织成 LLM 能高效利用的 prompt 上下文。

### 5.1 为什么需要（检索片段不能直接拼 prompt）

直接拼有四个问题：**重复**（两路召回 + AutoMerging 必重叠）、**无结构**（LLM 难定位）、**超 token**（top_n × chunk_size 可能超 LLM 上下文）、**无引用编号**（溯源断链）。context_builder 解决：**去重 → 排序 → token 截断 → 引用编号注入 → 格式化**。

### 5.2 去重（三层，由精确到模糊）

| 层 | 做什么 | 首版 |
|---|---|---|
| **① hash 去重** | 按 chunk_id / text hash 去完全相同（两路召回同一 chunk） | ✅ |
| **② 父子去重** | AutoMerging 回传父块后，同父块的子块已被父包含 → 去子块留父块 | ✅ |
| **③ 语义去重 / MMR** | 高度相似但非完全相同（overlap / 近重复）→ 相似度阈值去重或 MMR 选多样性 | ⏳ 后置 |

### 5.3 排序（lost-in-the-middle）

LLM 对上下文**头尾注意力强、中间弱**（"Lost in the Middle"）。三种策略：

| 策略 | 做法 | 首版 |
|---|---|---|
| **A 降序** | rerank 分数从高到低 | ✅（top_n=6 时上下文不长，影响有限） |
| **B 头尾高分** | 高分交替放头尾、低分中间 | ⏳（上下文变长时） |
| **C 文档分组** | 同文档 chunk 相邻（引用连贯） | ⏳ |

> rerank 分数还可做 token 预算分配（截断时高分优先保留）+ 引用置信度展示（UI）。

### 5.4 token 预算控制（tokenizer 选型）⭐

```
context_budget = LLM_MAX_CONTEXT_TOKENS
               - count_tokens(system_prompt) - count_tokens(query) - GENERATION_RESERVE
```

context_builder 按 budget 逐个累加 token（高分优先），超预算截断。

**tokenizer 选型**（影响预算准确性）：

| 方案 | 精度 | 首版 |
|---|---|---|
| GLM 专用 tokenizer（SDK 暴露时） | 最准（与 API 计费一致） | 核实 SDK 是否支持 |
| tiktoken + 15% buffer | 折中（中文与 GLM 有 5-15% 偏差，buffer 补偿） | ✅ 务实 |
| 粗估（字符 / 1.5） | 最差 | 不推荐 |

> 为什么不填满 LLM 上下文：① 接近上限时 LLM 质量退化（lost-in-the-middle 加剧）② 生成需留空间 ③ 输入 token 计费（填满 = 烧钱）。首版 `LLM_MAX_CONTEXT_TOKENS=6000`（保守）。

### 5.5 引用编号注入 + 三方契约 ⭐

每个检索片段注入**编号 + 元数据**：

```
[1]（智谱GLM技术概览.md, 第3页）GLM-4.5 是智谱 AI 开发的主力大语言模型...
[2]（模型对比.docx, 第1页）embedding-3 默认 2048 维...
```

**三方契约**（context_builder ↔ prompt ↔ CitationChecker 必须一致）：

| 方 | 职责 | 共享规范 |
|---|---|---|
| context_builder | 产出 `[N]（文档名,页码）文本` | 编号格式 + 元数据位置 |
| prompt 指令 | 告诉 LLM「论断后标 `[N]`」 | 编号格式绑定 |
| CitationChecker | 正则 `\[(\d+)\]` 提取 → 与上下文编号集合比对 | 解析格式绑定 |

> 改一方要同步三方（如格式从 `[N]` 改 XML 标签 → prompt + checker 同步改）。建议抽共享常量。

### 5.6 难点

| 难点 | 说明 |
|---|---|
| 去重不彻底 | overlap / 近重复，hash 去不掉（需语义去重/MMR） |
| token 计算精度 | tiktoken 与 GLM tokenizer 偏差 → 预算不准（buffer 补偿） |
| 排序策略冲突 | lost-in-the-middle（头尾高分）vs 引用连贯（文档分组）不能同时最优 |
| 空上下文 | 检索为空 → 不注入 → 生成层兜底「未找到」 |
| 父子混合 token | AutoMerging 后大父块（~1800 字）+ 小子块混合 → 预算分配难 |

### 5.7 进阶（首版全后置）

**① MMR 多样性**：不只看相关性（rerank 分数），还看多样性——`MMR_score = λ×relevance - (1-λ)×max_sim(d, selected)`，避免上下文全是高度相似内容，增加信息覆盖。适合综述/对比查询。

**② 多轮上下文压缩**：窗口截断（最简）→ 历史摘要（LLM 压缩旧轮）→ 选择性保留 → 历史检索。难点：历史幻觉累积 + 压缩损失 + token 分配（历史多了检索就少）。

**③ 动态上下文（query 感知）**：不同 query 类型（事实/综述/多跳）需不同 top_n / 去重 / token 分配。需查询路由判断 query 类型（同 §2 触发判据难题）。

### 5.8 agents_rag 落地

| 项 | 首版 | 说明 |
|---|---|---|
| 去重 | hash + 父子 | 精确；语义/MMR 后置 |
| 排序 | rerank 分数降序 | top_n 小时 lost-in-the-middle 影响有限 |
| token 预算 | 6000（tiktoken + buffer） | 保守，留足生成空间 |
| 截断 | 高分优先占预算 | 逐个累加 token，超预算停 |
| 引用编号 | `[N]（文档名,页码）文本` | 三方契约一致 |
| 格式 | 纯文本拼接 | 紧凑、省 token |
| 多轮 | ⏳ 后置 | 首版单轮 |

### 5.9 上下文构建层常见陷阱

| 陷阱 | 后果 |
|------|------|
| 不去重 | 重复内容浪费 token、干扰生成 |
| 无脑填满上下文 | 质量退化 + 成本高 |
| 截断不分优先级 | 高相关被截、低相关留下 |
| 无引用编号 / 编号与元数据脱节 | 溯源断链、CitationChecker 无法校验 |
| 父子不去重 | 父块 + 同父子块都在上下文 → 冗余 |
| 三方契约不同步 | context 改了格式但 prompt/checker 没跟 → 生成乱标、校验失败 |
| tokenizer 误差无 buffer | 实际 token 超预算 → API 报错 |

## 6. 生成（Generation）⭐

> 把上下文 + query 变成「带引用的可信回答」。prompt 工程是核心杠杆；置信度评估决定「敢不敢答」。

### 6.1 Prompt 工程（核心杠杆）

system prompt 四约束：

| 约束 | 作用 | 对抗 |
|------|------|------|
| **Grounding** | 仅基于上下文回答 | LLM 用自身知识编造（幻觉主源） |
| **强制引用** | 论断后标 `[N]` | 无来源论断（不可溯源） |
| **先结论后展开** | 结构化 | 用户先看答案 |
| **兜底** | 上下文不足 → 明确说「未找到」 | 硬编答案（最危险幻觉） |

prompt 是生成环节**最大可调杠杆**——同样的上下文/模型/温度，换 prompt 质量差 20-30%。但也是最不可复现的（换模型/数据最优解变）→ 评测驱动。

### 6.2 Grounding 可靠性（三重保障）

「仅基于上下文」靠三层叠加才接近可靠：

| 层 | 机制 | 可靠性 |
|---|---|---|
| **① prompt 约束** | system prompt 强指令 | ~80%（大部分遵守，非绝对） |
| **② 低温度** | T=0.1-0.3 减少采样发散 | 降低「创造性编造」概率 |
| **③ 后处理校验** | CitationChecker + faithfulness（§7） | 兜底 |

> 只靠 prompt grounding = 有 ~20% 漏网。真正的抗幻觉靠 §7 后处理校验。

### 6.3 温度 + 格式

**温度**：T→0 忠实（贪心），T→1 发散。事实问答 T=0.1-0.3；综述/对比 T=0.3-0.5。T=0 有时死板（该展开不展开），T=0.3 常优于 T=0。评测扫 T ∈ {0, 0.1, 0.3, 0.5} 看 faithfulness / relevancy。

**格式**：先结论后展开 + 分点（首版）；结构化/JSON mode 服务化阶段。top_p 可与温度配合控制发散（一般只调一个）。

### 6.4 置信度评估 + 拒答 ⭐⭐

#### 置信度的本质

「可信」包含三层：①上下文相关（检索准）+ ②答案忠实（没编造）+ ③引用可靠（来源命中）。不同信号只覆盖一层 → **每个信号都有盲区**。

#### 信号谱（6 方案）

| 方案 | 信号类型 | 反映 | 成本 | 盲区 |
|------|---------|------|------|------|
| ① rerank 分数 | 检索侧 | 上下文相关（①） | 免费 | 相关 ≠ 忠实 |
| ② CitationChecker 通过率 | 格式侧 | 引用命中（③） | 免费 | 命中 ≠ 内容支撑 |
| ③ logprob | 生成侧 | 模型自信 | 免费（需API） | 自信 ≠ 正确（hallucination 高 logprob） |
| ④ faithfulness | LLM judge | 答案忠实（②） | 1 LLM | judge 偏差 |
| ⑤ 自评 | LLM 自评 | 自评忠实 | 1 LLM | 过度自信 |
| ⑥ 一致性 | 多次采样 | 答案稳定 | N×生成 | 稳定 ≠ 正确 |

> **核心洞察**：没有任何单一信号可靠判断「可信」。rerank 管「找得准」、CitationChecker 管「标得对」、faithfulness 管「编没编」——三者正交互补。

#### 聚合方案（终极）

```python
confidence = w1 × normalize(rerank_score)    # 检索质量
           + w2 × citation_pass_rate          # 引用可靠性
           + w3 × faithfulness_score          # 内容忠实性（权重最高）
```

权重靠评测调；阈值在「拒答率」和「错误率」间平衡（企业场景倾向高阈值：宁可不答）。

#### 置信度在管线的位置 + 早期/后期拒答

```
②检索 → ③RRF → ④Rerank(rerank分数A) → ... → ⑦生成 → ⑧CitationChecker(引用率B) + faithfulness(C)
                                                              │
                                                        ⑨ 聚合(A+B+C)
                                                              │
                                                        ⑩ 拒答判断
```

| 判断窗口 | 节点 | 触发 | 价值 |
|---|---|---|---|
| **早期 1** | ②检索后 | 召回=0 | 直接拒答（**不生成**，省 LLM） |
| **早期 2** | ④Rerank 后 | top1 分数极低 | 有召回但都不相关（可选省生成） |
| **后期** | ⑧校验后 | citation/faithfulness/综合 低 | 生成完但不可信 |

#### 拒答返回设计

核心原则：**诚实 + 区分原因 + 引导行动 + 可选低置信展示**。

| 场景 | 返回消息 | 引导 |
|------|---------|------|
| 检索为空 | 「未找到与您问题相关的内容」 | 「确认文档已入库 / 换提问方式」 |
| 检索有但不相关 | 「未找到高度相关的内容」 | 「用更具体关键词」 |
| 忠实性低 | 「无法给出确切答案」 | 「补充上下文 / 拆分问题」 |
| 综合低置信 | 「未找到确切答案」 | 「换提问方式 / 确认文档已入库」 |

**结构化返回**（三态）：

| status | 含义 | answer |
|--------|------|--------|
| `answered` | 正常（置信高） | 完整回答 + 引用 |
| `low_confidence` | 低置信（灰色区） | 回答 + ⚠「仅供参考」标注 |
| `no_result` | 拒答 | null + message + suggestion |

> `low_confidence` 是中间态——高风险领域（医疗/法律）用 `no_result`（严格拒答）；低风险（内部知识库）用 `low_confidence`（展示 + 警告）。

### 6.5 进阶（首版全后置）

| 进阶 | 做法 | 价值 |
|------|------|------|
| **prompt 模板调优** | A/B 测试不同模板，评测驱动选最优 | 最大质量杠杆（±20-30%） |
| **Self-Check 修正循环** | 生成→校验→反馈修正→再校验 | faithfulness 自动提升 |
| **结构化输出** | JSON mode / function calling（answer+citations+confidence 分离） | UI/API 友好 |
| **流式生成** | 逐 token 返回（打字机效果） | 用户感知延迟降 |
| **多模型协作** | GLM-4.5 主生成 + Flash 辅助（改写/校验）+ 独立 judge | 成本分层 |
| **多轮** | 每轮重新检索 + 历史窗口/压缩 | 多轮对话 |

### 6.6 难点

| 难点 | 说明 |
|------|------|
| Grounding 不可靠 | prompt ~80%，需后处理（§7） |
| 引用标注质量 | 漏标/错标/过度标 → CitationChecker 兜底 |
| 置信度聚合 | 信号量纲不同 + 相关性 + 权重过拟合 |
| 拒答阈值标定 | 拒答率 vs 错误率权衡，需评测 |
| 延迟/成本 | 在线实时，Self-Check/多次采样成本敏感 |
| 多轮幻觉累积 | 历史错误当事实 |

### 6.7 agents_rag 落地

| 项 | 首版 | 说明 |
|----|------|------|
| 模型 | GLM-4.5 | 主力生成 |
| system prompt | 四约束（Grounding/引用/先结论/兜底） | 核心杠杆 |
| 温度 | 0.3 | 忠实优先 |
| 上下文注入 | context_builder 产出（带编号）→ system prompt |
| 引用 | 强制标注 `[N]` | 衔接 CitationChecker |
| 置信度 | 检索空兜底（早期拒答 1） | 聚合/低置信后置 |
| 多轮 | ⏳ 后置 | 首版单轮 |

### 6.8 生成层常见陷阱

| 陷阱 | 后果 |
|------|------|
| 无 Grounding 约束 | LLM 用自身知识编造 |
| 温度过高（0.7+） | 事实问答发散编造 |
| 不强制引用 | 回答无来源（不可溯源） |
| 检索空时不兜底 | LLM 自由发挥（幻觉最严重） |
| 置信度用单一信号 | 盲区漏判（如 logprob 自信≠正确） |
| 拒答不区分原因 | 统一「不知道」，用户/管理员无法定位问题 |

## 7. 引用溯源与抗幻觉 ⭐⭐

> 企业 RAG 与玩具 demo 的分水岭：**敢不敢亮来源、能不能在不知道时认怂**。本节不是独立管线步骤，而是贯穿生成后的「校验 + 溯源」层，把前面散落的抗幻觉机制系统化串联。

### 7.1 引用溯源（完整链路）

溯源是一条**贯穿全管线的元数据链**：

```
知识构建 → chunk metadata（doc_id / source / page / section_path / char_span）
    ↓ 检索命中子块，metadata 随之可用
上下文构建 → [N]（文档名, 页码）文本            ← 编号注入（§5.5 三方契约）
    ↓ 生成引用编号
生成 → 回答中标注 [N]                           ← prompt 强制引用（§6.1）
    ↓ 后处理
CitationChecker → [N] 命中校验 → 有效引用列表    ← 本节核心
    ↓
展示 → 文档名 + 页码 + 原文片段 + 可点击跳转     ← UI（后期）
```

**链路完整性**：任何一环元数据脱节 → 溯源断。知识构建保证 metadata 完整，§5.5 三方契约保证编号一致。`image_ref` 为图片描述回答溯源到原图铺路。

### 7.2 抗幻觉：纵深防御（5 层）⭐

幻觉不是靠单一机制解决——是**纵深防御**，每层拦截不同类型：

| 层 | 机制 | 拦截什么 | 位置 | 可靠性 |
|---|---|---|---|---|
| **第 1 层** | Prompt Grounding + 低温度（§6） | 大部分「用自身知识编造」 | 生成时 | ~80% |
| **第 2 层** | **CitationChecker**（编号校验） | 引用虚假（标了不存在的编号） | 生成后 | 格式可靠 |
| **第 3 层** | **faithfulness**（忠实性校验） | 编造（论断无上下文支撑） | 生成后（进阶） | 最直接，judge 有偏差 |
| **第 4 层** | 置信度拒答（§6.4） | 低可信回答（综合信号低） | 聚合后 | 多信号兜底 |
| **第 5 层** | **内容一致性校验** | 编号对但内容错（数值/方向偏差） | 进阶 | 最难，需 NLI/judge |

> 核心认知：第 1 层拦 80%，漏网的 20% 交给第 2-3 层后处理，残余交给第 4 层拒答。每层拦截**不同类型**的幻觉。

### 7.3 第 2 层：CitationChecker（核心实现）⭐

**基础校验（编号命中）**：

```python
def check_citations(answer_text, context_ids: set[int]):
    cited = set(int(m) for m in re.findall(r'\[(\d+)\]', answer_text))
    valid = cited & context_ids          # 有效引用
    invalid = cited - context_ids        # 无效引用（标了不存在的）
    # 漏标检测
    unsourced = [s for s in sentences if has_claim(s) and not has_citation(s)]
    pass_rate = len(valid) / max(len(cited), 1)    # 置信度信号 B
```

**三种偏差处理**：

| 偏差 | 检测 | 处理 |
|---|---|---|
| **错标**（[9] 但上下文只有 [1-6]） | `invalid = cited - context_ids` | 剔除 / 标记「引用无效」 |
| **漏标**（有事实但没标编号） | `unsourced` 列表 | 标记「无来源」 |
| **过度标**（每句都标含无关句） | 引用密度统计 | 标记冗余（不阻塞） |

### 7.4 第 3 层：faithfulness（忠实性校验）

| 维度 | CitationChecker | faithfulness |
|---|---|---|
| 校验什么 | 引用**编号**命中 | 论断**内容**有无上下文支撑 |
| 成本 | 免费（正则+集合） | 1 次 LLM（judge） |
| 检测 | 错标（不存在的编号） | 编造（无支撑论断） |
| 盲区 | 编号对但内容错 | judge 漏判 |

> **互补**：CitationChecker 校验「引用格式」、faithfulness 校验「内容忠实」。两者都过 = 引用可靠 + 内容可信。

**实现**：RAGAS faithfulness（LLM 逐句分解 + 比对上下文）或自建 judge prompt。judge 模型选 GLM-4.5（强推理）或独立 judge（避免「自己判自己」过度自信）。judge 偏差缓解：强模型 + 多次采样取多数。

### 7.5 第 5 层：内容一致性校验（进阶）

编号命中 ≠ 内容支撑。例：上下文 [1] 说「2048 维」，生成写「4096 维[1]」——编号对、数值错。CitationChecker 基础版检测不到。

- **NLI（自然语言推理）**：判断「引用原文 是否蕴含 论断」（entail / contradict / neutral）
- **LLM judge**：给定引用原文 + 论断 → 判断支撑关系
- 代价：每个引用一次 NLI/judge。首版不做，faithfulness 部分覆盖。

### 7.6 难点

| 难点 | 说明 |
|---|---|
| CitationChecker 只校验格式 | 编号命中 ≠ 内容支撑（需一致性/NLI，进阶） |
| faithfulness judge 偏差 | LLM 判断不完美（漏判/不一致）；成本/延迟 |
| 溯源链路完整性 | 任何一环元数据脱节 → 溯源断 |
| 「最后一公里」幻觉 | prompt + CitationChecker + faithfulness 都过了，但仍有微妙偏差（数值/方向） |
| 校验延迟/成本 | CitationChecker 免费；faithfulness + 一致性需额外 LLM |

### 7.7 agents_rag 落地

| 项 | 首版 | 后置 |
|---|---|---|
| CitationChecker（编号校验 + 剔除无效引用） | ✅ | — |
| 引用溯源（返回来源：文档名+页码+片段） | ✅ | — |
| 检索空兜底 + 无效引用标记 | ✅ | — |
| faithfulness | ⏳ | RAGAS 评测就绪后 |
| 内容一致性校验（NLI/judge） | ⏳ | 高价值但成本高 |
| 置信度聚合拒答 | ⏳ | §6.4 三信号 |
| UI 跳转原文 | ⏳ | Streamlit 阶段 |

### 7.8 引用溯源/抗幻觉层常见陷阱

| 陷阱 | 后果 |
|------|------|
| 只靠 prompt grounding（无后处理） | ~20% 幻觉漏网 |
| 不做 CitationChecker | 引用虚假、不可溯源 |
| 编号与元数据脱节 | 溯源断链（不知来源文档/页码） |
| 不做兜底（检索空硬编） | 幻觉最严重场景 |
| faithfulness 用同模型自判 | 过度自信（「自己判自己」偏差） |
| 只校验编号不校验内容 | 「编号对但数值错」漏检 |

## 8. 评估（RAGAS）⭐

> 没有评估的 RAG 是玄学。RAGAS 的价值不在「分数是 0.85」——在于**指标下降时能定位到具体环节**。

### 8.1 四指标深入（计算方法 + LLM judge 角色）

| 指标 | 计算方法 | LLM judge | 诊断环节 |
|------|---------|-----------|---------|
| **Context Recall** | `\|检索 ∩ 相关文档\| / \|相关文档\|` | 不需（集合比对） | 检索召回 + 构建分块 |
| **Context Precision** | 检索中相关比例 + 排序质量（precision@k / MRR） | 不需（需标注） | 检索 + Rerank |
| **Faithfulness** | 答案逐句分解 → 每句判断有无上下文支撑 → 忠实句占比 | **LLM 逐句 judge** | 生成 + 抗幻觉 |
| **Answer Relevancy** | 从答案反向生成 query → 与原 query 比对 | **LLM 反向生成** | 生成 + 改写 |

> **关键区分**：Recall/Precision **不需 LLM**（标注集 + 集合比对，可离线批量跑）；Faithfulness/Relevancy **需 LLM judge**（成本 + 偏差）。

### 8.2 评测集构建（最大瓶颈）⭐

没有评测集 = RAGAS 无法运行 = 调参靠玄学。

**结构**：`query + ground_truth（标准答案）+ relevant_chunk_ids（应召回）+ query_type`

**数据来源（质量递增）**：

| 来源 | 质量 | 适用 |
|------|------|------|
| 专家编写 | 偏差大（不反映真实用户） | 冷启动 |
| 从文档生成 Q-A | LLM 半自动，偏差中等 | 起步扩充 |
| **真实日志采样** | **最佳**（真实分布） | 成熟期 |

**「相关文档」标注最耗时**：对每个 query 在已索引 chunk 池里人工定位应召回的；多人标注 + 一致性检验（Cohen's kappa）。

**规模**：冷启动 30-50 条；成熟几百条。

> **与知识构建 recall@k 评测集共享**——标注「query + 相关 chunk」后，构建侧调 chunk_size（recall@k）和查询侧 RAGAS（Context Recall）共用。加 ground_truth → Faithfulness/Relevancy 也能跑。

### 8.3 指标归因（核心价值）⭐⭐

| 指标下降 | 问题环节 | 排查方向 |
|---------|---------|---------|
| **Context Recall ↓** | 检索召回 + 构建分块 | chunk_size / top_k / embedding 质量 / 分块策略 |
| **Context Precision ↓** | 检索 + Rerank | Rerank 模型 / RRF 参数 / top_n |
| **Faithfulness ↓** | 生成 + 抗幻觉 | prompt grounding / 温度 / CitationChecker |
| **Answer Relevancy ↓** | 生成 + 改写 | query 理解 / prompt 格式 / 改写方向 |

> 这是 RAGAS 区别于一般「端到端打分」的关键——**四指标分别诊断四环节**，不是笼统的「RAG 好不好」。

### 8.4 评测方法谱（多层互补）

| 方法 | 特点 | 成本 |
|------|------|------|
| **RAGAS 自动** | 快、可批量、依赖标准答案 + LLM judge | 中（judge） |
| **人工评测** | 最准、能评 RAGAS 覆盖不到的（流畅/完整） | 高人力 |
| **LLM-as-judge** | 折中（注意 judge 偏差） | 1 LLM/条 |
| **在线（A/B + 用户反馈）** | 最真实但稀疏/滞后 | 低（埋点） |

**多层评测**：RAGAS（自动回归）+ 人工（抽检）+ 在线（真实分布），互补。

### 8.5 评测陷阱

| 陷阱 | 后果 | 缓解 |
|------|------|------|
| 过拟合评测集 | 调参适配评测集 ≠ 线上好 | hold-out 测试集 |
| 偏离真实分布 | 专家编造 ≠ 用户真实 query | 真实日志采样 |
| 规模太小 | 统计不显著 | 扩充到几百 |
| 陈旧过时 | 文档更新但评测集没跟 | 定期更新 |
| judge 偏差 | faithfulness/relevancy 不完美 | 强模型 + 多次采样 |
| 标注不准 | recall/precision 算错 | 多人标注 + 一致性 |

### 8.6 进阶

- **参数网格自动化**：chunk_size × top_k × top_n × 温度 联合网格扫 → RAGAS 四指标 → 选最优组合
- **分类型评测**：按 query_type 分组看指标 → 某类差 → 针对性优化
- **CI 集成**：代码改动 → 自动跑 RAGAS → 指标下降告警（回归保护）
- **在线反馈闭环**：用户 👍👎 → 标注 → 扩充评测集 → 反哺调优

### 8.7 难点

| 难点 | 说明 |
|------|------|
| 评测集构建 | 标注「相关文档」最耗时；日志采样需去敏 |
| LLM judge 成本 | faithfulness/relevancy 每条 1 LLM |
| judge 偏差 | 漏判/不一致 |
| 归因不唯一 | 指标下降可能多环节共同导致 |
| 过拟合 | 适配评测集 ≠ 线上好 |

### 8.8 agents_rag 落地

| 项 | 首版 | 后置 |
|----|------|------|
| 评测集 | 30-50 条（专家编 + 日志采样） | 扩充到几百 |
| recall@k | ✅（构建侧已建） | — |
| RAGAS 4 指标 | ⏳（需 ground_truth + RAGAS 框架） | 评测集加标准答案后 |
| 参数网格自动化 | ⏳ | 联合网格 + CI |
| hold-out | ⏳ | 防过拟合 |
| 在线反馈 | ⏳ | 服务化阶段 |

> 首版：先建评测集（query + 相关 chunk）跑 recall@k（构建侧已就绪）；加 ground_truth 后跑 RAGAS 四指标。

### 8.9 评估层常见陷阱

| 陷阱 | 后果 |
|------|------|
| 无评测集 | 调参靠玄学、回归无保障 |
| 评测集太小 | 统计不显著 |
| 评测集偏离分布 | 过拟合、线上差 |
| 只看端到端不分环节 | 定位不了问题 |
| 不做 hold-out | 过拟合评测集 |
| judge 用弱模型 | faithfulness/relevancy 不准 |

## 9. 常见陷阱清单（避坑）

| 陷阱 | 后果 |
|------|------|
| 原始 query 直接检索 | 模糊 / 口语化 query 召回差 |
| 只用向量、无 BM25 | 专有名词 / 编号召回漏 |
| 检索即终、无 rerank | precision 上不去 |
| 上下文无脑拼接、无去重 / 预算 | token 浪费、lost-in-the-middle |
| 不做引用校验 | 模型乱标编号、可信度崩 |
| 无评估集 | 调参靠玄学、回归无保障 |
| 多轮把历史当事实 | 历史幻觉累积 |

## 10. 小结：查询管线配方（agents_rag）

```
用户问题 → (查询改写/HyDE 可选)
        → 向量召回 + BM25 召回 → RRF 融合 → Rerank 精排（默认智谱）
        → 上下文构建(去重 + token 预算 + 父子回传 + 引用编号)
        → GLM-4.5 生成(grounding + 强制引用)
        → 引用校验 → [带引用来源的回答]
        (RAGAS 贯穿评估)
```

核心取舍：**召回宽、精排严、上下文精、生成守、全程测**。检索与生成是质量主战场。
